# Skills 简体中文本地化规范

本规范用于 `D:\skills\skills-main` 中 skills 的中文本地化、中文增强和定稿流程。除非用户明确确认，任何翻译/增强草稿都不得覆盖源文件。

---

## 1. 工作原则

- **先出草稿，后定稿**：翻译或增强阶段一律新建 `*_zh-CN.enhanced.md` 草稿，不覆盖原文件。
- **定稿需确认**：用户说"定稿"后，先列出删除、重命名、索引同步、插件检查清单，并获得确认。
- **定稿方式固定**：确认后先删除原文件，再把已审核增强稿重命名为原文件名；不要复制草稿内容覆盖原文件。
- **索引必须同步**：定稿后同步 bucket README 和根 README 的描述；如果项目声明了插件入口，检查 `.Codex-plugin/plugin.json`。
- **当前合成版 README 暂不逐项同步**：`D:\skills\superpowers\README.md` 已不作为逐个 skill 的目录索引；单个 skill 定稿时不需要同步该 README，等整个项目汉化完毕后再统一整理目录索引。
- **不擅自清理**：删除草稿、源文件或多余文件前必须明确列清单并确认。
- **先确认审核意见，再决定修改**：收到任何审核意见后，先逐条说明是否认可及理由；只有在双方确认需要调整后，才修改、重构或定稿，不默认直接改文件。

## 2. 全局审查清单

每完成一个 skill 的汉化或增强，逐项检查：

1. **翻译腔**：逐句检查是否自然，避免直译和生硬搭配。
2. **可汉化术语**：识别裸露英文名词，按术语表统一处理。
3. **中英混排**：英文术语作为名词嵌入中文句子，不半句中文半句英文。
4. **行为等价与合理性**：不得改变原 skill 的核心行为；增强只能补边界、说明、检查和本土化提示。
5. **安全边界**：不自动执行危险操作，不替用户登录、推送、删除、改全局配置或修改外部系统。
6. **代码块/路径/命令保留**：JSON、bash、TypeScript、文件路径、CLI 命令、标签常量一律保持可执行/可引用。
7. name / description 规范 — name 保持 ASCII slug；description 同样通过翻译腔、中英混排、术语统一检查，2 句格式。必须包含**用户口吻触发短语**：以"用户说""用户提到""用户想""用于用户说"等句式，列出中文口语中可能说出的自然表达（如"收尾""帮我合并""代码写完了"），不能只有技术前提的场景描述。
8. **索引同步**：bucket README 与根 README 描述应和最终 `SKILL.md` 行为一致。
9. **中国本土化**：检查工具可用性、平台替代、语言默认、文档习惯和工作流适配（按第 3 节细化表逐项确认）。
10. **模板产出内容汉化**：最终写入用户文档、issue、评论正文的标题、固定提示句、占位说明和默认输出文本均应汉化；非最终产出的格式参考模板可保留英文但需说明可按团队语言调整。
11. **子目录全覆盖审查**：审查 skill 时必须连带检查子目录（`scripts/`、`references/`、`examples/` 等）内的所有文件，逐一判定是否需翻译、是否可保留英文及理由。代码级文件（`.sh`、`.js`、`.cjs`、`.html`、`.py` 等）默认不翻译，但浏览器/终端可见的固定英文文本必须指出位置并给出建议。prompt 模板、配置示例、参考指南等文档类子文件必须出草稿。

## 3. 中国本土化审查（细化版）

本土化不是把所有英文改成中文，而是确认 skill 在中文团队和中国网络环境下可用。  
请逐项自检：

### 3.1 工具可用性
- [ ] 依赖的 CLI、npm 包、GitHub/GitLab 认证等是否可能受网络影响？
  - 若受影响，**仅提示用户**按团队规范配置 registry、nrm、认证令牌或代理。
  - **禁止**自动修改用户全局配置，也不要主动给出会修改全局配置的命令，除非用户明确要求。

### 3.2 平台替代
- [ ] 是否兼容 GitLab 自托管、Gitee、Coding.net、Jira、Linear、本地 markdown 等非 GitHub 流程？
  - 若无法直接支持，是否提供了“Other/自定义流程”兜底方案（如手动导出数据、使用 webhook 转换）？

### 3.3 语言默认
- [ ] agent 生成的用户文档、PRD、CONTEXT.md、ADR 等是否默认适配中文？
  - 若示例用英文，是否明确说明“实际项目可使用中文或团队约定语言”？

### 3.4 文档习惯
- [ ] 是否默认使用 git‑based docs 路径？
  - 若用户使用语雀、飞书等外部文档，是否要求用户将流程描述为文本步骤（因为 skill 无法直接操作这些系统）？

### 3.5 工作流适配
- [ ] 是否保留人工确认、评审、HITL/AFK 区分、提交前检查等协作习惯？
  - 不得默认“一切全自动”。对于破坏性操作（如批量删除 issue），必须有用户明确确认步骤。

### 3.6 日期/数字/格式本土化（新增）
- [ ] 生成的文档中，日期是否为 `YYYY年MM月DD日` 或用户习惯格式？
- [ ] 千位分隔符是否使用逗号（中文科技文档兼容）或按项目约定？

## 4. 汉化策略优先级

| 优先级 | 策略 | 适用场景 |
|:--:|------|---------|
| P0 | 忠实翻译 | 纯工具型、配置型、行为敏感但可等价翻译的 skill |
| P0 | 中文增强 | 需要补充中文工作流、本土化提示、安全边界或示例的 skill |
| P1 | 英文原版 + 中文导读 | 上游同步成本高、术语容易漂移或行为极敏感的 skill |
| P2 | 新建 skill | 目标任务与已有 skill 明显不同，触发条件和工作流独立 |

> P1 策略仅在以下情况使用：上游每两周更新一次且 skill 包含复杂正则/DSL；或翻译后可能破坏语义等价性。

## 5. 术语统一表

| 英文 | 中文 | 规则 |
|------|------|------|
| repo | 仓库（repo） | 首次加注，后续 `仓库` |
| block | 配置块 | 用于写入配置的块；不要改 `## Agent skills` 字面内容 |
| section | 小节 | `Section A/B/C` 作为标签可保留 |
| seed template | 种子模板（seed template） | 首次加注 |
| glossary | 术语表（glossary） | 首次加注，后续 `术语表` |
| context | 上下文（context） | 首次加注，后续 `上下文`；`single-context` / `multi-context` 配置值不改 |
| lazily create | 按需创建 | 避免"懒创建" |
| stub | 桩文件（stub） | 首次加注，后续 `桩文件` |
| variant | 变体（variant） | 首次加注，后续 `变体` |
| maintainer | 维护者（Maintainer） | 首次加注 |
| devDependency | 开发依赖 | 必要时首次写 `开发依赖（devDependencies）` |
| smoke test | 冒烟测试（smoke test） | 不要译为"拷机测试" |
| issue | 任务项 / 问题 | issue tracker 场景可保留 `issue`；非技术上下文中可汉化 |

| ready for agent | 已准备好交给 AFK agent | 标签常量 `ready-for-agent` 保留 |

## 6. 不可汉化或谨慎汉化清单

以下类型保持英文或只在解释性文字中加中文说明：

- 产品/平台名：GitHub、GitLab、Gitee、Coding.net、Jira、Linear、Husky、Prettier、shoehorn。
- CLI 命令和 flag：`gh`、`glab`、`gh issue create`、`--label`、`-F json`。
- 文件名和路径：`AGENTS.md`、`CONTEXT.md`、`AGENTS.md`、`docs/agents/`、`docs/adr/`、`.scratch/`。
- 标签常量和配置值：`needs-triage`、`ready-for-agent`、`single-context`、`multi-context`。
- 格式名：`markdown`、`heredoc`、`dash-case`。
- XML/伪标签：`<prd-template>`、`<issue-template>` 等。
- JSON、TypeScript、Python、XML 等代码块内的键名、代码、命令和示例。`bash` 代码块中用于解释命令步骤的说明类注释可汉化；真正的命令、flag、路径、环境变量和参数值保持原样。JS/TS/Python/XML 等编程代码注释默认保留原文，除非该注释模拟最终用户可见文档或终端输出。
- 引用原文：如 `publish to the issue tracker` 这种被 skill 明确引用的原句。
- `issue tracker`、`triage labels` 等配置概念可在解释性句子中说明，但路径、模板名、引用原文中保持英文。

## 7. 模板产出内容判定

| 类型 | 处理 |
|------|------|
| 最终写入用户文档的 `##` 标题 | 应汉化，例如 PRD 的 `问题陈述`、`解决方案`、`用户故事` |
| issue/PRD 正文模板 | 标题、固定提示句、占位说明和默认输出文本均应汉化；示例格式可保留英文并说明可按团队语言调整 |
| 固定输出文本 | 应汉化，例如 `None - can start immediately` → `无 — 可以立即开始` |
| 写入 `AGENTS.md` / `AGENTS.md` 的结构标识 | 可保留英文，例如 `## Agent skills`、`### Issue tracker` |
| Markdown 示例正文 | 应汉化；即使示例位于代码块中，只要它模拟的是用户文档、issue、PRD、comment、README、CONTEXT.md 或 ADR 的正文，就应汉化标题、段落、列表、占位说明和默认输出文本 |
| 代码/命令示例 | 不动；仅保留真正的程序代码、CLI 命令、JSON/YAML/TOML 键名、文件路径、标签常量、API 名称和编程代码注释。`bash` 代码块中的说明类注释可汉化；JS/TS/Python/XML 等编程注释默认保留原文，除非它模拟最终用户可见文档或终端输出 |
| JSON/代码块内键名 | 不动 |
| XML/伪标签 | 不动 |
| 格式参考模板（非最终产出） | 如果模板展示的是 Markdown 文档正文，默认汉化正文内容；只有字段名、命令、路径、代码、标签常量等不可汉化内容保留英文。若保留英文示例，必须说明它是外部协议或上游格式要求 |

## 8. 草稿命名与多文件 skill

- 单文件 skill 草稿命名：`SKILL_zh-CN.enhanced.md`。
- 多文件 skill 草稿命名：`原文件名_zh-CN.enhanced.md`，例如 `domain_zh-CN.enhanced.md`、`ADR-FORMAT_zh-CN.enhanced.md`。
- 草稿中的相对链接应保持指向定稿后的原文件名，不要改成 `_zh-CN.enhanced.md`。
- 多文件 skill 定稿时逐个列出原文件和增强稿的删除/重命名对应关系。
- **依赖文件处理**：若增强稿引用了未汉化的图片、脚本等，**不得修改引用路径**，保持原文件名。

## 9. 定稿流程（含验证）

用户要求"定稿"时执行以下流程：

1. 列出将删除的原文件（完整路径）。
2. 列出将重命名的增强稿及目标文件名。
3. 列出需要同步的 README / plugin 文件。
4. 明确这是破坏性操作，等待用户确认。
5. 确认后删除原文件，再重命名增强稿。
6. 同步 bucket README 和根 README。
7. 如果项目声明了插件入口（存在 `.Codex-plugin/plugin.json`），检查该文件是否已有对应 skill，按项目规则补齐。未声明插件入口的项目跳过此步。
8. **验证步骤（新增）**：
   - 在对话中报告验证方式和结果。
   - 若存在原 skill 提供的核心测试用例，说明是否运行以及结果。
   - 若无可测试用例，至少说明一个典型用户场景的人工冒烟检查结果。
   - 只有当用户明确要求，或项目已有验证日志机制时，才写入 `.validation.log`。
9. 校验目录中无增强稿残留。

> 默认不创建 `.backup` 备份文件；如用户要求备份，先列出备份文件名和后续清理方式并确认。

## 10. README 与 plugin 同步规则

- `engineering/`、`productivity/`、`misc/` 中的 skill 必须出现在根 `README.md` 和对应 bucket README 中。
- bucket README 的描述应简洁说明最终 `SKILL.md` 的能力。
- 根 README 的 Reference 描述应与 bucket README 同步或略详细。
- 如果 `description` 或核心行为说明发生变化，定稿前必须重审 bucket README 和根 README，不直接复用旧索引句。
- 如果项目声明了插件入口（存在 `.Codex-plugin/plugin.json`），该文件必须包含上述三个 bucket 中的正式 skill。
- `personal/`、`in-progress/`、`deprecated/` 不应出现在根 README 或 plugin 中。

## 11. 审核输出建议

每次提交草稿后，建议让用户按以下维度审核：

- 合理性：是否保留原 skill 行为，是否新增了不该有的能力。
- 翻译腔：是否有直译、生硬、冗余、中文不自然。
- 术语：是否符合术语表，是否有可汉化裸词。
- 本土化：CLI、平台、网络、文档语言、团队流程是否兼容。
- 坑位：是否会覆盖用户内容、误改远端、创建重复标签、默认错误平台。
- 索引：定稿后 README/plugin 是否需要同步。

## 12. 自动化检查（Linter）要求

为实现“机器守门”，每个草稿提交前应优先按以下规则人工检查。若项目已有 linter、用户要求创建 linter，或后续已在仓库中加入自动化脚本，再运行自动化检查；不要为了满足本节而临时创建脚本。

### 12.1 必检项
- [ ] **中英混排空格**：人工检查英文术语嵌入中文句子时是否自然；若使用自动正则，应豁免反引号、路径、命令、文件名、标签常量、frontmatter、Markdown 链接、代码块和产品名。
- [ ] **不可汉化路径/文件名被改**：检测如 `Codex\.md`、`docs/adr/` 等是否被替换（模糊匹配） → 报错。
- [ ] **术语表一致性**：扫描草稿中的英文术语（从术语表读取），若既未按表翻译又未出现在“不可汉化清单”中 → 警告。
- [ ] **代码块内键名被汉化**：检测 JSON key 是否变成中文（如 `"描述":`） → 警告。
- [ ] **代码注释处理边界**：检查 `bash` 说明类注释是否可按中文团队需要汉化；JS/TS/Python/XML 等编程注释是否未被误改，除非它模拟最终用户可见文档或终端输出。
- [ ] **Markdown 示例正文未汉化**：检查代码块中的 `markdown` / `md` 示例；如果它模拟最终用户文档、issue/comment、PRD、CONTEXT.md、ADR 或 README 正文，标题、段落、列表和默认输出文本都应汉化，不能只用“示例可按团队语言调整”兜底。
- [ ] **模板占位符被修改**：检测 `<[a-z\-]+>` 类标签是否内容改变 → 警告。

### 12.2 推荐工具实现
- 如用户要求创建自动化检查，可使用 Python 脚本或 `grep` + `sed` 组合，并按项目约定放到 `scripts/linter/` 目录。
- 输出格式为：`文件名:行号: 错误类型: 具体内容`。

### 12.3 豁免机制
- 若确有特殊情况需忽略某条规则，可在草稿头部添加 `<!-- lint ignore rule-name -->` 注释，并附简短原因，由审查者二次确认。

## 常见错误反模式清单

| ID | 错误示例（❌） | 正确示例（✅） | 违反规范 | 解释 |
|----|----------------|----------------|----------|------|
| A01 | “懒创建文件夹” | “按需创建文件夹” | 术语表 `lazily create` | 直译生硬，规范要求“按需创建” |
| A02 | “请使用GitHub来clone仓库” | “请使用 GitHub 来 clone 仓库” | 中英混排 | 英文前后缺少空格 |
| A03 | 将 `AGENTS.md` 写成 `克劳德.md` | 保持 `AGENTS.md` | 不可汉化清单 | 文件名必须原样 |
| A04 | 将 JSON 中的 `"description"` 译为 `"描述"` | 保持 `"description"` | 不可汉化清单 | 代码块内键名不得翻译 |
| A05 | 将 `<prd-template>` 改成 `<prd-模板>` | 保持 `<prd-template>` | 不可汉化清单 | XML 伪标签必须原样 |
| A06 | 未提示用户配置代理，直接调用 `gh` 命令 | 先检测网络，若失败则提示“请设置 `GITHUB_TOKEN` 或代理” | 本土化 3.1 | 国内用户可能无法直接访问 GitHub |
| A07 | 将 `None - can start immediately` 译为“无” | “无 — 可以立即开始” | 模板产出汉化 | 固定输出文本应完整汉化 |
| A08 | 将 `issue` 一律译成“问题”，导致 issue tracker 场景混淆 | 技术上下文保持 `issue` 或“任务项” | 术语表 | 需区分 issue tracker 与普通问题 |
| A09 | description 只有场景描述，缺少用户口吻触发词 → "实现已完成、测试已通过，并且需要决定如何合入时使用。" | 同时包含场景描述和用户口吻触发词 → "实现已完成、测试已通过，并且需要决定如何合入时使用。用户说"收尾""完成分支""代码写完了"时也可触发。" | §2.7 触发词规范 | 用户说"收尾、帮我合并"等口语时无法匹配，导致 skill 不触发 |

> 本清单会持续更新，每次审查发现新的高频错误时加入。
