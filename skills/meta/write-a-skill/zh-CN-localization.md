# 简体中文 Skill 本地化规范

本规范用于创建、翻译、增强和审查简体中文 agent skill。目标是在保持原 skill 行为等价的前提下，让中文团队可读、可用、可维护。

除非用户明确确认，不要在翻译或增强阶段覆盖源文件。默认先产出草稿，再经用户审核后定稿。

## 工作原则

- **先出草稿，后定稿**：翻译或增强阶段新建 `*_zh-CN.enhanced.md` 草稿，不直接覆盖源文件。
- **保持行为等价**：忠实翻译不得改变原 skill 的核心触发条件、执行流程和安全边界。
- **增强需标明边界**：中文增强只能补充本土化提示、中文工作流、安全检查、示例和输出格式，不应引入不相关能力。
- **定稿需确认**：涉及删除、覆盖、重命名、迁移、推送或修改外部系统时，先列清单并等待用户确认。
- **定稿方式固定**：确认后先删除原文件，再把已审核增强稿重命名为原文件名；不要复制草稿内容覆盖原文件。
- **不擅自清理**：删除草稿、源文件或多余文件前，必须说明将删除什么、为什么删除。
- **先确认审核意见，再修改**：收到审核意见后，先逐条判断是否认可及理由，再根据确认结果修改。

## 汉化策略

| 策略 | 适用场景 | 要求 |
|---|---|---|
| 忠实翻译 | 原 skill 行为敏感、流程明确，只需中文化 | 保持行为等价，不新增流程 |
| 中文增强 | 需要适配中文团队、本土工具或协作习惯 | 可补充边界、说明、检查和示例 |
| 英文原版 + 中文导读 | 上游频繁更新、术语容易漂移或翻译风险高 | 保留英文执行文件，新增中文导读 |
| 新建 skill | 目标任务与已有 skill 明显不同 | 独立定义触发条件、流程和资源 |

## 全局审查清单

每完成一个 skill 的汉化或增强，逐项检查：

1. **翻译腔**：逐句检查是否自然，避免直译和生硬搭配。
2. **术语统一**：识别裸露英文名词，按术语表统一处理。
3. **中英混排**：英文术语作为名词嵌入中文句子，不半句中文半句英文。
4. **行为等价与合理性**：不得改变原 skill 的核心行为；增强只能补边界、说明、检查和本土化提示。
5. **安全边界**：不自动执行危险操作，不替用户登录、推送、删除、改全局配置或修改外部系统。
6. **代码块/路径/命令保留**：JSON、bash、TypeScript、文件路径、CLI 命令、标签常量一律保持可执行、可引用。
7. **`name` / `description` 规范**：`name` 保持 ASCII slug；`description` 使用自然中文，包含用户口吻触发短语。
8. **索引同步**：如果项目有 README、目录索引、插件清单或 marketplace 元数据，定稿时同步描述；正式发布目录应进入索引，草稿、个人、废弃或实验目录不应进入发布索引。
9. **中国本土化**：检查工具可用性、平台替代、语言默认、文档习惯和工作流适配。
10. **模板产出内容汉化**：最终写入用户文档、issue、评论正文的标题、固定提示句、占位说明和默认输出文本均应汉化。
11. **子目录全覆盖审查**：连带检查 `scripts/`、`references/`、`examples/` 等资源文件。

## Description 规范

`description` 是 agent 判断是否加载 skill 的主要依据。中文 skill 的 `description` 应清楚说明能力和触发条件，但不要把完整工作流塞进去。

要求：

- `name` 保持 ASCII slug。
- `description` 使用自然中文。
- 中文本地化场景必须采用两句结构：第一句说明能力，第二句说明触发条件；确需例外时，必须说明原因。
- 必须包含用户口吻触发短语，例如“用户说……”“用户想……”“用户提到……”。
- 不要只有抽象场景描述。
- 保留 skill name、slash command、文件名、路径、包名、API 名称和 CLI 命令。
- 不要总结完整流程，避免 agent 只照 `description` 做而跳过正文。

好例子：

```text
实现已完成、测试已通过，并且需要决定如何合入、开 PR、保留或清理分支时使用。用户说“收尾”“完成分支”“代码写完了”“帮我合并”时也可触发。
```

坏例子：

```text
用于完成开发流程。
```

## 中英混排

英文术语作为名词嵌入中文句子时，应自然、清晰，避免半句中文半句英文。

差：

```text
帮用户 create 一个 skill with proper structure。
```

好：

```text
帮助用户创建结构清晰的 skill。
```

保留英文标识：

- 产品和平台名：GitHub、GitLab、Gitee、Coding.net、Jira、Linear、OpenCode、Claude Code、Codex、Husky、Prettier。
- CLI 命令和 flag：`gh`、`glab`、`npm install`、`--label`、`-F json`。
- 文件名和路径：`AGENTS.md`、`CLAUDE.md`、`CONTEXT.md`、`docs/adr/`、`.scratch/`。
- 标签常量和配置值：`needs-triage`、`ready-for-agent`、`single-context`、`multi-context`。
- 格式名：`markdown`、`heredoc`、`dash-case`。
- XML/伪标签：`<prd-template>`、`<issue-template>`。
- JSON/YAML/TOML key、代码符号、API 名称。

## 术语统一

| 英文 | 推荐中文 | 说明 |
|---|---|---|
| repo | 仓库 | 首次可写“仓库（repo）” |
| block | 配置块 | 用于写入配置的块 |
| section | 小节 | `Section A/B/C` 作为标签可保留 |
| seed template | 种子模板 | 首次可写“种子模板（seed template）” |
| glossary | 术语表 | 首次可写“术语表（glossary）” |
| context | 上下文 | 配置值如 `single-context` 不翻译 |
| lazily create | 按需创建 | 避免“懒创建” |
| stub | 桩文件 | 首次可写“桩文件（stub）” |
| variant | 变体 | 首次可写“变体（variant）” |
| maintainer | 维护者 | 首次可写“维护者（Maintainer）” |
| devDependency | 开发依赖 | 必要时写“开发依赖（devDependencies）” |
| smoke test | 冒烟测试 | 不译为“拷机测试” |
| issue | issue / 任务项 | issue tracker 场景可保留 `issue` |
| demo | 演示代码 | 首次可写“演示代码（demo）” |
| reporter | 提交者 | 用于 issue 作者、反馈者、报告者 |
| agent brief | agent brief | 可解释为“给 AFK agent 的执行规格” |
| out of scope | 范围外 | 目录名 `.out-of-scope/` 保留 |
| acceptance criteria | 验收标准 | 模板产出内容中应汉化 |
| blocked by | 阻塞项 | 模板产出内容中应汉化 |
| ready for agent | 已准备好交给 AFK agent | 标签常量 `ready-for-agent` 不翻译 |

## 模板产出内容

凡是最终会写入用户文档、issue、PRD、评论、README、CONTEXT.md 或 ADR 的正文，默认应中文化。

应汉化：

- Markdown 标题。
- 固定提示句。
- 占位说明。
- 默认输出文本。
- issue / PRD / 评论模板正文。
- 示例文档正文。

不应汉化：

- 代码。
- CLI 命令。
- JSON/YAML/TOML key。
- 文件路径。
- 标签常量。
- API 名称。
- XML/伪标签。
- 外部协议要求的固定字段。

示例：

```text
None - can start immediately
```

应译为：

```text
无 — 可以立即开始
```

## 中国本土化审查

本土化不是把所有英文改成中文，而是确认 skill 在中文团队和中国网络环境下可用。

检查项：

- 依赖的 CLI、npm 包、GitHub/GitLab 认证等是否可能受网络影响。
- 若受网络影响，只提示用户按团队规范配置 registry、认证令牌或代理。
- 不自动修改用户全局配置。
- 不主动给出会修改全局配置的命令，除非用户明确要求。
- 是否兼容 GitLab 自托管、Gitee、Coding.net、Jira、Linear 和本地 markdown 流程。
- 无法直接支持时，是否提供手动导出、webhook 转换或自定义流程兜底。
- 用户文档、PRD、CONTEXT.md、ADR、issue 正文和评论是否默认使用中文或团队约定语言。
- 若用户使用语雀、飞书等外部文档系统，要求用户提供文本内容或流程说明，不假设 agent 能直接操作外部系统。
- 保留人工确认、评审、HITL / AFK 区分和提交前检查。
- 日期格式优先使用 `YYYY年MM月DD日` 或团队约定格式。
- 数字分隔按项目约定；未约定时，中文科技文档可使用逗号千分位。

## 子目录审查

审查一个 skill 时，必须同时检查其子目录和资源文件。

需要检查：

- `scripts/`
- `references/`
- `examples/`
- prompt 模板
- 配置示例
- 参考文档
- 图片或附件引用

处理原则：

- 代码文件默认不翻译。
- 浏览器、终端或用户可见的固定英文文本需要指出，并判断是否应汉化。
- prompt 模板和参考指南属于文档类内容，通常需要汉化或新增中文草稿。
- 引用图片、脚本或资源文件时，不要随意改路径。
- 草稿中的相对链接应指向定稿后的原文件名，不要指向 `_zh-CN.enhanced.md`。

## 安全边界

禁止默认执行以下操作：

- 删除文件。
- 覆盖源文件。
- 重命名正式文件。
- 修改全局配置。
- 替用户登录。
- 推送远端。
- 创建或删除远端 issue / PR / 标签。
- 批量迁移外部系统数据。

如确需执行，必须先列出：

- 将修改或删除的文件。
- 将重命名的文件及目标路径。
- 将同步的 README、索引、配置或插件文件。
- 风险说明。
- 验证方式。

等待用户明确确认后再执行。

## 定稿流程

用户明确要求“定稿”时：

1. 列出将删除或覆盖的原文件。
2. 列出将重命名的草稿及目标文件名。
3. 列出需要同步的索引、README、插件或配置文件。
4. 明确这是破坏性操作。
5. 等待用户确认。
6. 确认后先删除原文件，再把已审核草稿重命名为原文件名；不要复制草稿内容覆盖原文件。
7. 完成后检查是否仍有草稿残留。
8. 报告验证方式和结果。

如果原 skill 有测试，说明是否运行以及结果。

如果没有可运行测试，至少说明一个典型用户场景的人工冒烟检查结果。

## 索引与插件同步

如果项目维护 README、目录索引、插件清单、manifest、marketplace 元数据或包管理配置，skill 定稿后必须检查是否需要同步。

同步原则：

- 正式发布目录中的正式 skill 应出现在对应索引或插件清单中。
- 草稿、个人、实验、废弃、迁移中或范围外目录不应进入发布索引或插件清单。
- 如果 `description`、名称、触发范围或核心行为说明发生变化，定稿前必须重审相关 README、索引和插件描述，不直接复用旧描述。
- 如果项目明确声明某个索引暂不逐项同步，遵循项目级说明，并在定稿报告中注明该豁免。
- 同步时保留路径、包名、配置 key、标签常量和平台要求，不为“汉化”而改坏可执行配置。

## 自动化检查建议

提交草稿前应人工检查：

- 中英混排是否自然。
- 路径、文件名、命令和配置 key 是否被误译。
- 术语是否统一。
- JSON/YAML/TOML key 是否保持原样。
- Markdown 示例正文是否按最终产出规则汉化。
- `<...>` 伪标签是否保持原样。
- 是否包含真实密钥、token、生产 URL 或个人隐私。
- 是否引入自动删除、推送、登录或修改全局配置的行为。

如果项目已有 linter 或用户要求创建 linter，可以添加自动化检查。不要为了形式临时创建复杂脚本。

### 豁免机制

若确有特殊情况需要忽略某条检查规则，可在草稿头部添加注释，并写明原因：

```markdown
<!-- lint ignore rule-name: 简短说明原因 -->
```

豁免必须由审查者二次确认。不要用豁免掩盖可修复问题，也不要为省事批量豁免。

## 常见错误

| 错误 | 正确做法 |
|---|---|
| “懒创建文件夹” | “按需创建文件夹” |
| “请使用GitHub clone仓库” | “请使用 GitHub clone 仓库” |
| 将 `AGENTS.md` 翻译成中文文件名 | 保持 `AGENTS.md` |
| 将 JSON key `"description"` 译为 `"描述"` | 保持 `"description"` |
| 将 `<prd-template>` 改成 `<prd-模板>` | 保持 `<prd-template>` |
| 未确认网络和认证就调用远端 CLI | 先说明依赖和风险 |
| 将 `None - can start immediately` 只译为“无” | 译为“无 — 可以立即开始” |
| `description` 只有抽象场景，没有用户口吻触发词 | 增加“用户说……”“用户想……”等自然触发短语 |

## 交付建议

交付草稿时，建议提示用户按以下维度审查：

- 行为是否保持原 skill 等价。
- 是否有翻译腔或生硬表达。
- 术语是否统一。
- 是否符合中文团队工作流。
- 是否误改代码、路径、命令、配置 key 或标签常量。
- 是否存在覆盖、删除、推送、登录等安全风险。
- 定稿后是否需要同步 README、索引、插件或配置。
