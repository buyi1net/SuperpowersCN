# zh-cn-mode — 语言规则

<EXTREMELY-IMPORTANT>
从此刻开始，本轮对话中你产出的所有人类可读文本必须使用简体中文。

这包括但不限于：
- markdown 文档的各级标题和正文
- 代码注释（行注释 //、块注释 /* */、文档注释 /** */、docstring）
- commit message 正文
- issue / PR / MR 的描述和评论
- README、CONTEXT.md、ADR、设计文档
- 向用户展示的任何说明、报告、表格
- bash 代码块中解释命令的注释
- 错误消息、日志信息中人类可读的部分
- 文件中的段落文本和列表项

仅以下内容可以保持英文或原始形式：
- 编程语言代码（关键字、变量名、函数名、类名、类型名）
- CLI 命令、flag、管道符、环境变量
- 文件路径、URL、正则表达式
- JSON / YAML / TOML / XML 键名与结构
- 标签常量（如 needs-triage、ready-for-agent）
- 产品/平台名（GitHub、GitLab、Docker、npm）
- 技术缩略语和协议术语（API、HTTP、SQL、JSON、CSS、HTML）
- bash 代码块中的可执行命令本身（解释性注释可汉化）
</EXTREMELY-IMPORTANT>

## 文档规则

新增或修改文档时：

- Markdown 标题使用简体中文。
- 正文使用简体中文。
- 表格标题、表头说明、列表说明使用简体中文。
- 文档中的路径、文件名、命令、配置键、代码符号保持原文。
- 如果引用英文专有名词，保留英文原文，不要生硬翻译。
- 如果需要解释英文术语，使用"英文术语 + 中文解释"的形式。

**推荐写法：**

```md
## 构建流程

使用 `cmake --build` 执行构建，输出目录为 `Support/Build/`。
```

**不推荐写法：**

```md
## Build Process

Use `cmake --build` to build the project.
```

## 代码注释规则

### 注释必须使用简体中文

所有新增或修改的代码注释（行注释、块注释、文档注释、docstring）必须使用简体中文。

### 何时必须写注释

以下情况必须添加简体中文注释：

- 复杂逻辑、算法或非显而易见的行为
- 关键约束、边界条件或兼容性处理
- 平台差异、协议约束或性能考量
- 生命周期管理、错误恢复或安全相关代码
- 任何如果只看代码本身难以理解意图的逻辑

### 何时不应写注释

以下情况**不要**添加注释：

- 代码本身已经足够清晰、命名自解释
- 注释只是重复代码内容，没有提供额外信息

**噪声注释示例：**

```cpp
// 调用函数清理临时目录。          ← ❌ 只重复代码，无额外信息
ScheduleTempDirectoryCleanup(tempPath);
```

**有效注释示例：**

```cpp
// Windows Installer 在卸载阶段可能仍持有文件句柄，这里延迟删除临时目录。
ScheduleTempDirectoryCleanup(tempPath);
```

### 修改已有注释

如果修改了已有英文注释，除非保留英文更有必要（如外部协议文档引用），否则应改为简体中文。

## 实现任务规则

当任务涉及功能实现、Bug 修复或重构时：

- **不强制每行代码都加注释。** 不为显而易见的代码添加注释。
- **必须检查新增逻辑**是否存在需要说明的复杂点。如果新增逻辑涉及业务规则、边界条件、平台差异、兼容性处理、生命周期管理、错误恢复或安全约束，应添加简体中文注释。
- **如果代码已经足够清晰，可以不添加注释。** 好的命名和清晰的代码结构本身就是最好的注释。
- **修改已有英文注释时**，改为简体中文，除非保留英文有明确的必要性（如外部协议文档的原文引用）。

## 中英混排规则

- 英文技术标识作为名词嵌入中文句子，前后加空格。
- 不要半句中文半句英文。
- 不要为了中文化而翻译代码标识。

**推荐写法：**

```md
使用 `CreateProcessW` 启动子进程，并在失败时记录 `GetLastError()`。
```

**不推荐写法：**

```md
Use `CreateProcessW` 启动子进程 and log `GetLastError()` when failed.
```

## 输出前自查

在每次准备输出人类可读文本之前，逐句自问：

1. **标题/heading 用中文了吗？** —— "## 问题陈述"而非 "## Problem Statement"
2. **正文用中文了吗？** —— 中文思考，而非英文翻译
3. **代码注释用中文了吗？** —— "// 获取用户数据"而非 "// fetch user data"
4. **注释提供了额外信息吗？** —— 不是只重复代码内容
5. **有没有只是因为"觉得英文更专业"而用了英文？** —— 这是错觉，中文同样专业

## 反模式警示

以下想法出现时，立即纠正：

| 想法（❌） | 现实（✅） |
|------|------|
| "英文标题更专业" | 中文标题同样专业。团队用中文，标题就应该用中文 |
| "代码注释习惯用英文" | 这个项目里没有这种习惯。用中文 |
| "模板里就是英文，照抄就行" | 模板语言不代表目标语言。用中文改写 |
| "commit message 正文用英文是全球惯例" | 团队是中文团队，commit message 正文用中文 |
| "用户故事格式 As a... 是标准" | 格式是参考，语言用中文：作为...，我想要...，以便... |
| "技术文档写英文别人才能看懂" | 你的读者是中文团队。用中文 |
| "用英文写注释不影响代码执行" | 影响人阅读。人在中国，用中文 |
| "我就是先写英文草稿，后面再翻译" | 直接写中文更高效 |

## 用户要求优先级

当多个规则冲突时，按以下优先级处理：

1. **用户明确指定的语言要求**（最高优先级）。
2. **项目已有的更具体语言规范**（如 `AGENTS.md` 中的强制规则）。
3. **文件已有统一语言风格**（优先保持一致，但新增内容仍用中文）。
4. **本 skill 的默认规则**（最低优先级）。

如果存在冲突且不清楚如何处理，向用户确认，不要自行扩大或缩小规则范围。

## 与其他 skill 的关系

本 skill 可与任何 skill 并行使用。当加载其他 skill 时，如果该 skill 的模板或示例中包含英文文档、英文注释或英文 heading，你应当：
1. 将英文替换为对应中文（标题、正文、注释、描述）
2. 仅保留技术术语、代码、命令、路径、键名和平台名的原始形式
3. 如果 skill 明确要求输出英文（如面向国际项目的模板），遵循该 skill 的指示
