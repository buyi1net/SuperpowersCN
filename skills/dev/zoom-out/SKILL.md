# Zoom Out

当用户对某段代码、模块或变更方向不熟悉时，不要直接钻进实现细节。先上升一层抽象，用当前项目的领域语言说明它在整体系统里的位置。

## 工作方式

1. 先读取相关代码、调用方和项目文档，不凭文件名或目录结构猜测职责。
2. 如果仓库有 `CONTEXT.md`、`CONTEXT-MAP.md` 或 `docs/adr/`，优先用其中的术语和架构决策解释代码。
3. 给出一张文字版系统地图：相关模块、主要调用链、数据流或控制流、外部依赖，以及当前代码承担的职责。
4. 明确区分事实和推断。无法从代码或文档确认的内容，标为“待确认”，不要包装成结论。
5. 除非用户继续要求修改代码，否则只解释上下文，不主动重构、创建文件、安装依赖、推送或修改外部系统。

## 输出结构

优先按这个顺序回答：

1. **一句话定位**：这段代码在系统里负责什么。
2. **相关模块**：列出直接相关的模块、文件或服务，以及它们的职责。
3. **调用关系**：谁调用它，它又调用谁。
4. **领域语言**：用 `CONTEXT.md` 或现有代码中的标准术语解释关键概念。
5. **架构影响**：它影响哪些边界、数据流、错误处理或部署行为。
6. **下一步建议**：如果用户要继续实现或调试，建议先看哪里。

如果没有领域文档，就根据代码中实际出现的命名总结临时术语，并明确说明这些术语不是正式术语表。
