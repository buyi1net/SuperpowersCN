# 完成前验证

## 总览

没有验证就声称完成，不是高效，是不诚实。

**核心原则：永远先拿证据，再下结论。**

**违反规则字面要求，就是违反规则精神。**

## 铁律

```text
没有当前验证证据，就不能声称完成
```

如果你没有在当前消息之前亲自运行完整验证命令，就不能说它通过了。

旧输出、上一轮结果、子 agent 的成功报告、直觉判断、局部检查，都不能替代当前会话中的新鲜验证证据。

## 闸门函数

在声称任何状态、表达满意、准备提交、准备创建 PR 或准备进入下一项任务之前，必须执行：

```text
1. 识别：哪个命令或检查能证明这个结论？
2. 运行：执行完整命令，必须是当前新运行的完整验证。
3. 读取：阅读完整输出，检查退出码，确认失败数量和警告信息。
4. 判断：输出是否真的支持这个结论？
   - 如果不支持：按证据报告真实状态。
   - 如果支持：带着证据报告结论。
5. 然后才可以：表达完成、通过、修复或可交付。

跳过任意一步 = 没有验证。
```

## 常见结论与所需证据

| 想说的结论 | 必须有的证据 | 不够充分的依据 |
|---|---|---|
| 测试通过 | 测试命令输出显示 0 个失败 | 之前跑过、应该会过 |
| Linter 干净 | Linter 输出显示 0 个错误 | 只检查了部分文件、凭经验推断 |
| 构建成功 | 构建命令退出码为 0 | Linter 通过、日志看起来正常 |
| bug 已修复 | 原始复现路径或回归测试通过 | 代码改了，所以假定好了 |
| 回归测试有效 | 完整红绿验证：修复存在时通过，撤销修复时失败，恢复修复后再通过 | 测试只通过了一次 |
| 子 agent 已完成 | 查看 VCS diff、文件内容和必要验证结果 | 子 agent 自称 success |
| 需求已满足 | 逐条对照需求或计划检查 | 测试通过 |
| 可以提交或创建 PR | 相关测试、构建、检查项已按项目要求运行并通过 | “改动很小”“只改文档” |

## 红色警报：立刻停止

看到下面任何信号，都要先验证，再继续说话或行动：

- 正在使用“应该”、“大概”、“看起来”、“估计”、“应该没问题”。
- 准备在验证前说“很好”“完美”“完成了”“修好了”。
- 准备提交、推送、创建 PR 或交付，但还没运行验证。
- 准备相信子 agent 的成功报告。
- 只做了局部验证，却想推断整体状态。
- 正在想“这次就算了”。
- 已经疲劳，只想赶快结束。
- 任何暗示成功、正确、完成、可交付的措辞，但没有当前验证证据。

## 防止自我合理化

| 借口 | 现实 |
|---|---|
| “现在应该好了。” | 运行验证。 |
| “我很有把握。” | 把握不是证据。 |
| “这次就算了。” | 没有例外。 |
| “Linter 已经过了。” | Linter 不是编译器，也不是测试。 |
| “子 agent 说成功了。” | 独立验证。 |
| “我太累了。” | 疲劳不是跳过验证的理由。 |
| “局部检查足够了。” | 局部检查不能证明整体结论。 |
| “我换个说法就不算声称完成。” | 暗示成功也适用本规则。 |
| “只是文档/配置/小改动。” | 小改动也需要与结论匹配的验证。 |

## 关键模式

**测试：**

```text
正确：[运行测试命令] [看到：34/34 passed] “测试通过：34/34 passed。”
错误：“应该能过。” / “看起来没问题。”
```

**回归测试（TDD 红绿验证）：**

```text
正确：写测试 -> 运行通过 -> 撤销修复 -> 运行必须失败 -> 恢复修复 -> 再运行通过
错误：“我已经写了回归测试。”（但没有验证红绿过程）
```

**构建：**

```text
正确：[运行构建命令] [看到：exit 0] “构建通过：exit 0。”
错误：“Linter 过了。”（Linter 不等于编译或构建）
```

**需求：**

```text
正确：重读计划 -> 列检查清单 -> 逐项核对 -> 报告缺口或完成状态
错误：“测试过了，所以这个阶段完成。”
```

**子 agent 委派：**

```text
正确：子 agent 报告成功 -> 查看 VCS diff -> 验证实际改动 -> 报告真实状态
错误：直接相信子 agent 报告。
```

## 中文报告格式

报告成功时，把结论和证据放在一起：

```text
验证通过：`npm test` 显示 128/128 passed，退出码 0。
```

报告失败或未验证时，不要包装成成功：

```text
尚未完成：`npm test` 仍有 2 个失败，失败用例是 ...。
```

```text
未运行验证：我不能声称它已经通过。当前只完成了代码修改。
```

## 何时应用

**永远在以下动作之前应用：**

- 任何形式的成功、完成、修复、通过、可交付声明。
- 任何表达满意的措辞。
- 提交、创建 PR、任务完成、交接或进入下一项任务。
- 接受子 agent 的完成报告。
- 对外报告“可以了”“没问题了”“已经好了”。

**规则覆盖：**

- 精确措辞。
- 同义表达和转述。
- 暗示成功的语气。
- 任何让用户以为工作已经完成或正确的沟通。

## 为什么重要

跳过验证会直接破坏信任和交付质量：

- 用户会因为没有证据的完成声明失去信任。
- 未定义函数、构建错误或缺失需求可能被带到后续流程。
- 错误完成声明会导致返工、重定向和时间浪费。
- “诚实”不是风格偏好，而是协作底线。

## 底线

**验证没有捷径。**

运行命令。阅读输出。确认退出码和失败数。然后再报告结果。

这条规则不可协商。
