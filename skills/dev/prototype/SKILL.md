# 原型

原型是**用来回答一个问题的可丢弃代码**。问题决定原型形态。

## 选择分支

先识别要回答哪个问题，可以从用户提示、周边代码判断；如果用户在线，也可以直接询问：

- **“这段逻辑 / 状态模型感觉对吗？”** -> [LOGIC.md](LOGIC.md)。构建一个很小的交互式终端应用，让状态机跑过那些纸面上难以推理的场景。
- **“这个应该长什么样？”** -> [UI.md](UI.md)。在单一路由上生成几个差异明显的 UI 变体，通过 URL 查询参数和底部悬浮切换栏切换。

两个分支会产出非常不同的东西；选错分支会让整个原型都白做。如果问题确实含糊且联系不上用户，就根据周边代码选择更匹配的分支（后端模块 -> 逻辑；页面或组件 -> UI），并在原型顶部写明这个假设。

## 两个分支都适用的规则

1. **从第一天起就是可丢弃代码，并清楚标记。** 原型代码应靠近它未来真正会被使用的位置（放在它要验证的模块或页面旁边），这样上下文清楚；但命名必须让随手读代码的人看出它是原型，不是生产代码。对于可丢弃 UI 路由，遵循项目已有路由约定；不要发明新的顶层结构。
2. **一个命令即可运行。** 使用项目现有任务运行器支持的方式，例如 `pnpm <name>`、`python <path>`、`bun <path>` 等。用户必须能不用思考就启动它。
3. **默认不持久化。** 状态保存在内存中。持久化通常是原型要检查的对象，不应成为原型依赖。如果问题明确涉及数据库，使用临时数据库或本地文件，并用清晰的 `PROTOTYPE — wipe me` 名称标记。
4. **跳过打磨。** 不写测试（仅限原型），不做超过“能跑起来”所需的错误处理，不抽象。目标是快速学到东西，然后删除。
5. **暴露状态。** 每次操作后（逻辑原型）或每次切换变体后（UI 原型），打印或渲染完整相关状态，让用户看到发生了什么变化。
6. **完成后删除或吸收。** 原型回答完问题后，要么删除它，要么把已验证的决策折入真实代码；不要让它在仓库里腐烂。

## 完成后

原型唯一值得保留的是**答案**。把答案和它要回答的问题记录到耐久位置，例如 commit message、ADR、issue，或原型旁边的 `NOTES.md`。如果用户在线，这可以是一段快速对话；如果用户不在线，就留下占位，让他们（或下一轮的你）在删除原型前补上结论。

## 中国本土化注意事项

- **依赖与网络**：优先使用项目既有工具链和依赖；不要为了原型自动修改 registry、代理、认证或全局配置。确需新增依赖或访问外网时，先说明原因并征求用户确认。
- **多平台兼容**：原型不绑定特定托管平台，可用于 GitHub、GitLab、Gitee、Coding.net 或本地项目环境。
- **产出语言**：`NOTES.md`、commit message 默认使用中文；代码注释遵循项目约定。
