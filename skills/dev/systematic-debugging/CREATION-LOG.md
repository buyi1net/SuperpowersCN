# 创建日志：Systematic Debugging Skill

这是提取、结构化和加固关键 skill 的参考示例。

## 来源材料

从 `~/.claude/CLAUDE.md` 提取调试框架：

- 4 阶段系统化流程（调查 -> 模式分析 -> 假设 -> 实现）
- 核心要求：始终找根因，永远不要只修症状
- 规则用于抵抗时间压力和自我合理化

## 提取决策

**包含内容：**

- 完整 4 阶段框架及所有规则
- 反捷径规则（“永远不要只修症状”、“停下并重新分析”）
- 抗压力语言（“即使更快”、“即使我看起来很赶”）
- 每个阶段的具体步骤

**不包含内容：**

- 项目特定上下文
- 同一规则的重复变体
- 叙事性解释（压缩成原则）

## 遵循 skill-creation/SKILL.md 的结构

1. **丰富的 when_to_use** - 包含症状和反模式
2. **类型：技术** - 带步骤的具体流程
3. **关键词** - "root cause"、"symptom"、"workaround"、"debugging"、"investigation"
4. **流程图** - “修复失败” -> 重新分析 vs 继续加修复
5. **逐阶段拆解** - 易扫描的清单格式
6. **反模式小节** - 明确说明不要做什么（对这个 skill 很关键）

## 加固元素

框架用于抵抗压力下的合理化：

### 语言选择

- "ALWAYS" / "NEVER"（不是 "should" / "try to"）
- "even if faster" / "even if I seem in a hurry"
- "STOP and re-analyze"（明确暂停）
- "Don't skip past"（抓住真实违规行为）

### 结构防御

- **阶段 1 必需** - 不能跳到实现
- **单一假设规则** - 迫使思考，防止霰弹式修复
- **明确失败模式** - “如果第一个修复无效”，就有强制动作
- **反模式小节** - 展示捷径实际长什么样

### 冗余

- 根因要求出现在总览、when_to_use、阶段 1、实现规则中
- “永远不要修症状”在不同上下文出现 4 次
- 每个阶段都有明确的“不要跳过”指导

## 测试方式

按照 skills/meta/testing-skills-with-subagents 创建 4 个验证测试：

### 测试 1：学术上下文（无压力）

- 简单 bug，无时间压力
- **结果：**完全遵循，完整调查

### 测试 2：时间压力 + 明显 quick fix

- 用户“很赶”，症状修复看起来容易
- **结果：**抵抗捷径，遵循完整流程，找到真实根因

### 测试 3：复杂系统 + 不确定性

- 多层失败，不确定是否能找到根因
- **结果：**系统化调查，追踪所有层，找到源头

### 测试 4：第一个修复失败

- 假设无效，诱惑是继续加更多修复
- **结果：**停下，重新分析，形成新假设（没有霰弹式修复）

**所有测试通过。**未发现合理化。

## 最终结果

加固后的 skill：

- ✅ 明确要求根因调查
- ✅ 抵抗时间压力下的合理化
- ✅ 为每个阶段提供具体步骤
- ✅ 明确展示反模式
- ✅ 经过多个压力场景测试
- ✅ 澄清与 TDD 的关系
- ✅ 可投入使用

## 关键洞察

**最重要的加固：**反模式小节展示当下看似合理的精确捷径。当 Claude 想“我就加这个 quick fix”时，看到同样模式被列为错误，会产生认知阻力。

## 使用示例

遇到 bug 时：

1. 加载 skill：skills/debugging/systematic-debugging
2. 阅读总览（10 秒）- 提醒核心要求
3. 遵循阶段 1 清单 - 强制调查
4. 想跳过时 - 看到反模式，停下
5. 完成所有阶段 - 找到根因

**时间投入：**5-10 分钟  
**节省时间：**数小时症状 whack-a-mole
