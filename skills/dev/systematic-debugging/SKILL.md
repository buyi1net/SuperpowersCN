# 系统化调试

## 总览

随机修复浪费时间，还会制造新 bug。快速补丁会掩盖底层问题。

**核心原则：修复前必须先找到根因。只修症状就是失败。**

**违反流程字面要求，就是违反调试精神。**

## 铁律

```text
没有先做根因调查，就不要修复
```

如果还没完成阶段 1，就不能提出修复方案。

## 何时使用

任何技术问题都用：

- 测试失败
- 生产 bug
- 异常行为
- 性能问题
- 构建失败
- 集成问题

**尤其在这些情况使用：**

- 时间压力大（紧急情况会诱人乱猜）
- “就一个 quick fix”看起来很明显
- 你已经试过多个修复
- 上一个修复没用
- 你没有完全理解问题

**不要因为这些原因跳过：**

- 问题看起来简单（简单 bug 也有根因）
- 你很赶（赶时间必然返工）
- 负责人要求“现在就修”（系统化比乱撞更快）

## 四个阶段

进入下一阶段前，必须完成当前阶段。

### 阶段 1：根因调查

**尝试任何修复前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 它们经常包含准确解法
   - 完整阅读堆栈
   - 记录行号、文件路径、错误码

2. **稳定复现**
   - 能可靠触发吗？
   - 精确步骤是什么？
   - 每次都会发生吗？
   - 如果无法复现 -> 收集更多数据，不要猜

3. **检查最近变更**
   - 哪些变更可能导致这个问题？
   - Git diff、最近提交
   - 新依赖、配置变更
   - 环境差异

4. **在多组件系统中收集证据**

   **当系统包含多个组件时（CI -> build -> signing，API -> service -> database）：**

   **提出修复前，先添加诊断信息：**
   ```text
   对每个组件边界：
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 验证环境/配置是否正确传递
     - 检查每一层的状态

   运行一次，收集证据，确认哪里断了
   然后分析证据，找出失败组件
   然后调查该组件
   ```

   **示例（多层系统）：**
   ```bash
   # 第 1 层：工作流
   echo "=== Secrets available in workflow: ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # 第 2 层：构建脚本
   echo "=== Env vars in build script: ==="
   env | grep IDENTITY || echo "IDENTITY not in environment"

   # 第 3 层：签名脚本
   echo "=== Keychain state: ==="
   security list-keychains
   security find-identity -v

   # 第 4 层：实际签名
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **这会揭示：**哪一层失败（secrets -> workflow ✓，workflow -> build ✗）

5. **追踪数据流**

   **当错误出现在调用栈深处：**

   阅读本目录中的 `root-cause-tracing.md`，使用完整的反向追踪技术。

   **快速版：**
   - 坏值从哪里来？
   - 谁用坏值调用了这里？
   - 持续向上追踪，直到找到源头
   - 在源头修复，不要在症状处修补

### 阶段 2：模式分析

**修复前先找模式：**

1. **寻找可工作的例子**
   - 在同一代码库中找相似的可工作代码
   - 哪些相似代码是正常的？

2. **对照参考实现**
   - 如果要实现某个模式，完整阅读参考实现
   - 不要扫读；逐行读
   - 先完全理解模式，再应用

3. **识别差异**
   - 可工作代码和坏代码有什么不同？
   - 列出每个差异，再小也算
   - 不要假设“那个不可能有影响”

4. **理解依赖**
   - 它需要哪些其他组件？
   - 需要哪些设置、配置、环境？
   - 它依赖哪些假设？

### 阶段 3：假设与测试

**科学方法：**

1. **形成单一假设**
   - 清楚说明：“我认为 X 是根因，因为 Y”
   - 写下来
   - 具体，不要含糊

2. **最小化测试**
   - 做能测试假设的最小变更
   - 一次只改一个变量
   - 不要一次修多个东西

3. **继续前先验证**
   - 有效？是 -> 阶段 4
   - 无效？形成新假设
   - 不要在原修复上继续叠补丁

4. **不知道时**
   - 说“我不理解 X”
   - 不要假装知道
   - 请求帮助
   - 继续研究

### 阶段 4：实现

**修根因，不修症状：**

1. **创建失败测试用例**
   - 最简单复现
   - 可以的话写自动化测试
   - 没有测试框架就写一次性测试脚本
   - 修复前必须有
   - 使用 `superpowers:test-driven-development` skill 编写合格的失败测试

2. **实现单一修复**
   - 处理已识别的根因
   - 一次只做一个变更
   - 不做“顺手改进”
   - 不捆绑重构

3. **验证修复**
   - 测试现在通过了吗？
   - 其他测试没坏吗？
   - 问题真的解决了吗？

4. **如果修复无效**
   - 停下
   - 计数：已经试过几个修复？
   - 如果 < 3：回到阶段 1，用新信息重新分析
   - **如果 >= 3：停下，质疑架构（见第 5 步）**
   - 没有架构讨论，不要尝试第 4 个修复

5. **如果 3 个以上修复失败：质疑架构**

   **架构问题信号：**
   - 每个修复都暴露不同位置的新共享状态/耦合/问题
   - 修复需要“大规模重构”才能落地
   - 每个修复都会在别处制造新症状

   **停下，质疑基本前提：**
   - 这个模式本身成立吗？
   - 我们是不是“只是因为惯性才坚持它”？
   - 应该重构架构，还是继续修症状？

   **尝试更多修复前，先和你的 human partner 讨论。**

   这不是假设失败；这是架构错了。

## 红色警报 - 停下并遵循流程

如果发现自己在想：

- “先 quick fix，之后再查”
- “改一下 X 看看”
- “多改几处再跑测试”
- “跳过测试，我手动验证”
- “大概是 X，我先修”
- “我没完全懂，但这个也许能行”
- “模式是 X，但我换种方式适配”
- “主要问题有这些：[没有调查就列修复]”
- 在追踪数据流前提出方案
- **“再试一个修复”**（已经试过 2 个以上）
- **每个修复都暴露别处的新问题**

**这些都意味着：停下。回到阶段 1。**

**如果 3 个以上修复失败：**质疑架构（见阶段 4.5）。

## 你的 human partner 发出的错误信号

**注意这些重定向：**

- “这件事没发生吗？” - 你未经验证就假设了
- “这能显示出……吗？” - 你本该先加证据收集
- “别猜了” - 你在没有理解前提下提修复
- “Ultrathink this” - 质疑基本前提，不只是症状
- “我们卡住了吗？”（沮丧） - 你的方法没效果

**看到这些：**停下。回到阶段 1。

## 常见合理化

| 借口 | 现实 |
|---|---|
| “问题简单，不需要流程。” | 简单问题也有根因。简单 bug 用流程也很快。 |
| “紧急，没时间走流程。” | 系统化调试比猜测-试错更快。 |
| “先试这个，再调查。” | 第一个修复会定下模式。从一开始就做对。 |
| “确认修复有效后再写测试。” | 未测试的修复站不住。先写测试才能证明。 |
| “一次改多个地方省时间。” | 无法隔离哪个变更有效，还会制造新 bug。 |
| “参考太长，我按模式改一下。” | 只理解一半必然出 bug。完整阅读。 |
| “我看到问题了，先修。” | 看到症状不等于理解根因。 |
| “再试一个修复。”（2 次以上失败后） | 3 次以上失败 = 架构问题。质疑模式，不要再修。 |

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|---|---|---|
| **1. 根因** | 读错误、复现、查变更、收集证据 | 理解 WHAT 和 WHY |
| **2. 模式** | 找可工作例子、对比 | 识别差异 |
| **3. 假设** | 形成理论、最小测试 | 确认假设或得到新假设 |
| **4. 实现** | 创建测试、修复、验证 | bug 解决，测试通过 |

## 当流程显示“没有根因”

如果系统化调查发现问题确实是环境、时序或外部因素导致：

1. 你已经完成流程
2. 记录调查过什么
3. 实现合适处理（重试、超时、错误信息）
4. 添加监控/日志，方便以后调查

**但** 95% 的“没有根因”都是调查不完整。

## 支持技术

这些技术属于系统化调试，位于本目录：

- **`root-cause-tracing.md`** - 沿调用栈向后追踪 bug，找到原始触发点

- **superpowers:verification-before-completion** - 声称修复成功前验证修复确实有效

## 中国本土化注意事项

- 如果复现、测试、构建或依赖安装因为 registry、代理、镜像、认证或网络失败而卡住，只报告失败和证据；不要自动修改全局 registry、代理、认证令牌或 shell 配置。
- 如果问题涉及 GitHub、GitLab、Gitee、Coding.net、Jira、Linear、CI 或远端环境，先确认平台、权限和认证状态；不可用时输出手动检查步骤，不替用户登录或改远端配置。
- 如果团队把排查记录放在语雀、飞书等外部文档系统，而当前环境无法直接操作，只输出 markdown 草稿或文本步骤，让用户同步。
- 日期、日志和排查记录默认使用中文项目习惯；需要写报告时可用 `YYYY年MM月DD日`，除非项目已有格式。

## 真实影响

来自调试会话：

- 系统化方法：15-30 分钟修复
- 随机修复方法：2-3 小时乱撞
- 首次修复成功率：95% vs 40%
- 引入新 bug：接近 0 vs 常见
