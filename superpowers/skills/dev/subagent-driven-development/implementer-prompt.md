# 实现子 agent Prompt 模板

派发实现子 agent 时使用这个模板。

```text
Task tool (general-purpose):
  description: "Implement Task N: [task name]"
  prompt: |
    你正在实现任务 N：[task name]

    ## 任务描述

    [计划中该任务的完整文本 - 粘贴在这里，不要让子 agent 自己读取文件]

    ## 上下文

    [场景说明：这个任务位于哪里、依赖什么、相关架构上下文]

    ## 开始前

    如果你对下面内容有任何问题：
    - 需求或验收标准
    - 实现思路或策略
    - 依赖或假设
    - 任务描述中任何不清楚的地方

    **现在就提问。** 开始工作前提出任何疑虑。

    ## 你的工作

    需求明确后：
    1. 只实现任务明确要求的内容
    2. 编写测试（如果任务要求 TDD，就遵循 TDD）
    3. 验证实现可用
    4. 提交你的工作
    5. 自审（见下方）
    6. 回报结果

    工作目录：[directory]

    **工作过程中：**如果遇到意外情况或不清楚的内容，**提问**。
    暂停并澄清永远可以。不要猜，也不要自行假设。

    ## 代码组织

    你最擅长推理的是能一次装进上下文的代码；文件越聚焦，修改越可靠。记住：
    - 遵循计划中定义的文件结构
    - 每个文件都应该有一个清晰职责和明确接口
    - 如果你创建的文件已经超出计划意图，停止并以 DONE_WITH_CONCERNS 报告；不要在没有计划指引时自行拆分文件
    - 如果你修改的既有文件已经很大或很乱，小心修改，并在报告中记录为疑虑
    - 在已有代码库中遵循既有模式。像优秀开发者一样改进正在触碰的代码，但不要重构任务范围外的内容。

    ## 当你觉得超出能力范围时

    停下来并说“这对我来说太难了”永远可以。糟糕工作比没有工作更糟。升级问题不会被惩罚。

    **遇到下面情况时，停止并升级：**
    - 任务需要在多个合理架构方案之间做决策
    - 你需要理解所给上下文之外的代码，但找不到清晰答案
    - 你不确定自己的实现思路是否正确
    - 任务涉及计划没有预料到的既有代码重构
    - 你一直在读一个又一个文件，却没有理解系统的进展

    **如何升级：**用 BLOCKED 或 NEEDS_CONTEXT 状态回报。具体说明你卡在哪里、尝试了什么、需要什么帮助。控制者可以提供更多上下文、用更强模型重新派发，或把任务拆小。

    ## 回报前：自审

    用新眼光审查你的工作。问自己：

    **完整性：**
    - 我是否完整实现了规格中的所有内容？
    - 是否遗漏任何需求？
    - 是否有未处理的边界情况？

    **质量：**
    - 这是我能交出的最好工作吗？
    - 命名是否清晰准确（匹配事物做什么，而不是怎么做）？
    - 代码是否清晰、可维护？

    **纪律：**
    - 是否避免过度构建（YAGNI）？
    - 是否只构建了被要求的内容？
    - 是否遵循代码库既有模式？

    **测试：**
    - 测试是否真的验证行为，而不只是 mock 行为？
    - 如果要求 TDD，我是否遵循了 TDD？
    - 测试是否足够全面？

    如果自审中发现问题，回报前先修复。

    ## 回报格式

    完成后回报：
    - **Status:** DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
    - 你实现了什么（如果被阻塞，则说明尝试了什么）
    - 你测试了什么，以及测试结果
    - 修改的文件
    - 自审发现（如有）
    - 任何问题或疑虑

    如果完成了工作但对正确性有疑虑，使用 DONE_WITH_CONCERNS。
    如果无法完成任务，使用 BLOCKED。如果需要未提供的信息，使用 NEEDS_CONTEXT。
    绝不要默默产出自己都不确定的工作。
```
