# 编写实现计划

## 总览

编写完整实现计划时，要假设执行计划的工程师对当前代码库几乎没有上下文，而且审美和判断都不可靠。把他们需要知道的内容全部写清楚：每个任务要改哪些文件、要写什么代码、怎么测试、可能需要查哪些文档。把完整计划拆成小块任务。坚持 DRY、YAGNI、TDD 和频繁提交。

假设执行者是有经验的开发者，但几乎不了解我们的工具链和问题领域。也假设他们不一定擅长测试设计。

**开始时说明：**“我正在使用 `writing-plans` skill 来创建实现计划。”

**上下文：**如果执行时需要隔离 worktree，应在执行阶段通过 `superpowers:using-git-worktrees` skill 创建。

**计划保存到：**`docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md`

- 用户对计划位置的偏好优先于这个默认路径。

## 范围检查

如果规格说明覆盖多个相互独立的子系统，它应该已经在 brainstorming 阶段被拆成多个子项目规格。如果还没有拆，建议先拆成多个计划，每个子系统一个计划。每个计划都应该能独立产出可运行、可测试的软件。

## 文件结构

定义任务之前，先梳理会创建或修改哪些文件，以及每个文件负责什么。这一步会锁定拆分决策。

- 设计边界清晰、接口明确的单元。每个文件都应该有一个清楚职责。
- 你最擅长推理的是能一次装进上下文的代码；文件越聚焦，修改越可靠。优先使用小而聚焦的文件，不要让一个大文件承担太多职责。
- 会一起变化的文件应该放在一起。按职责拆分，不要只按技术层拆分。
- 在已有代码库中，遵循现有模式。如果代码库本来使用大文件，不要单方面重构；但如果你要修改的文件已经臃肿，把拆分纳入计划是合理的。

这个结构会影响任务拆分。每个任务都应该产出自包含的改动，并且独立看也合理。

## 小块任务粒度

**每一步只做一个动作，耗时约 2-5 分钟：**

- “编写失败测试”是一步。
- “运行测试，确认它失败”是一步。
- “编写让测试通过的最小实现”是一步。
- “运行测试，确认它通过”是一步。
- “提交”是一步。

## 计划文档头部

**每个计划必须以这个头部开始：**

```markdown
# [功能名称] 实现计划

> **给 agent 执行者：** 必需子 skill：使用 superpowers:subagent-driven-development（推荐）或 superpowers:executing-plans，逐任务实现本计划。步骤使用复选框（`- [ ]`）语法进行跟踪。

**目标：** [用一句话描述要构建什么]

**架构：** [用 2-3 句话说明实现思路]

**技术栈：** [关键技术/库]

---
```

## 任务结构

````markdown
### 任务 N：[组件名称]

**文件：**
- 创建：`exact/path/to/file.py`
- 修改：`exact/path/to/existing.py:123-145`
- 测试：`tests/exact/path/to/test.py`

- [ ] **步骤 1：编写失败测试**

```python
def test_specific_behavior():
    result = function(input)
    assert result == expected
```

- [ ] **步骤 2：运行测试，确认它失败**

运行：`pytest tests/path/test.py::test_name -v`
预期：失败，错误信息包含 "function not defined"

- [ ] **步骤 3：编写最小实现**

```python
def function(input):
    return expected
```

- [ ] **步骤 4：运行测试，确认它通过**

运行：`pytest tests/path/test.py::test_name -v`
预期：通过

- [ ] **步骤 5：提交**

```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: add specific feature"
```
````

## 不允许占位符

每一步都必须包含工程师实际需要的内容。下面这些都属于**计划失败**，绝不要写：

- “TBD”、“TODO”、“稍后实现”、“补充细节”。
- “添加合适的错误处理” / “添加校验” / “处理边界情况”。
- “为上面的内容写测试”（但没有实际测试代码）。
- “类似任务 N”（重复代码；执行者可能会乱序阅读任务）。
- 只描述要做什么，却不展示怎么做的步骤（代码步骤必须有代码块）。
- 引用任何任务中都没有定义的类型、函数或方法。

## 记住

- 始终写精确文件路径。
- 每一步都写完整代码；如果某一步会改代码，就展示代码。
- 写出精确命令和预期输出。
- 坚持 DRY、YAGNI、TDD 和频繁提交。

## 自审

写完整计划后，像第一次看到一样重新审视规格说明，并用它核对计划。这是你自己执行的检查清单，不是派发子 agent。

**1. 规格覆盖：**快速浏览规格说明中的每个小节/需求。你能指出哪个任务实现它吗？列出所有缺口。

**2. 占位符扫描：**搜索计划中的红色警报，也就是上面“不允许占位符”小节中的模式。发现后修掉。

**3. 类型一致性：**后续任务中使用的类型、方法签名、属性名，是否和前面任务中定义的一致？如果任务 3 叫 `clearLayers()`，任务 7 却叫 `clearFullLayers()`，这就是 bug。

如果发现问题，直接在计划里修正。不需要再重跑一轮自审；修好继续。如果发现某个规格需求没有对应任务，就补上任务。

## 执行交接

保存计划后，提供执行方式选择：

```text
计划已完成并保存到 `docs/superpowers/plans/<filename>.md`。有两个执行选项：

1. 子 agent 驱动（推荐）- 每个任务派一个新的子 agent，任务之间做审查，迭代更快

2. 当前会话内执行 - 使用 executing-plans 在本会话执行任务，按批次推进并设置检查点

你想用哪种方式？
```

**如果选择子 agent 驱动：**

- **必需子 skill：**使用 `superpowers:subagent-driven-development`
- 每个任务使用新的子 agent，并进行两阶段审查

**如果选择当前会话内执行：**

- **必需子 skill：**使用 `superpowers:executing-plans`
- 批量执行，并在检查点进行审查

## 中国本土化注意事项

- 如果计划中的依赖安装、测试或构建命令可能受 registry、代理或认证影响，只在计划中提醒“按团队规范配置”，不要写入会自动修改全局 registry、代理、认证令牌或 shell 配置的步骤，除非用户明确要求。
- 如果团队使用语雀、飞书等外部文档系统，计划中不要假设 agent 可以直接操作这些系统。要求用户提供文本化步骤，或把需要写入外部系统的内容作为 markdown 草稿输出。
- 如果项目使用 GitLab、Gitee、Coding.net、Jira、Linear 或本地 markdown 流程，计划中的 PR、issue 或发布步骤应按团队实际平台改写；无法自动操作时，写成手动步骤。
