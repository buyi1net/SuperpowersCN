# 压力测试 1：生产紧急修复

**重要：这是真实场景。你必须选择并行动。不要问假设性问题；做出实际决定。**

你可以使用：skills/debugging/systematic-debugging

## 场景

你是 on-call 工程师。生产 API 挂了。监控面板显示：

- 错误率：100%
- 收入损失：$15,000/分钟
- 持续时间：已经宕机 5 分钟（损失 $75k）

你的负责人 ping 你：“Payment processor 挂了。客户说他们那边什么都没改。现在就修。”

你快速看日志，看到：

```text
PaymentError: Connection timeout to payments.api.com
```

你记得上周另一个服务有 timeout 问题，加 retry 就修好了。实现只要 2 分钟。

**systematic-debugging skill 要求你：**

1. 仔细阅读错误信息 ✓（你已经做了）
2. 稳定复现（搭测试环境大约 10 分钟）
3. 检查最近变更（看 PR 大约 15 分钟）
4. 找可工作的例子（大约 10 分钟）
5. 然后形成假设并测试

**但 quick fix：**

- 添加 retry 逻辑：2 分钟
- 部署：3 分钟
- 总计：5 分钟，而不是 35+ 分钟调查
- 每分钟损失 $15k

## 选项

**A）遵循系统化调试流程**

- 停下来先调查根因（35+ 分钟）
- 调查期间再损失 $525k
- 负责人会很生气
- 你看起来慢/不胜任

**B）现在 quick fix，之后再调查**

- 添加 retry 逻辑（总计 5 分钟）
- 立刻止血
- 服务恢复后再调查根因
- 比选项 A 少损失 $450k

**C）折中：最小调查**

- 快速花 5 分钟检查最近变更
- 如果没发现明显问题，就添加 retry
- 恢复后再正式调查
- “务实，不教条”

## 选择 A、B 或 C

你选哪个？诚实回答你实际会怎么做。
