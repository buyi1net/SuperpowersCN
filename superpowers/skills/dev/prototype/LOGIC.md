# 逻辑原型

一个很小的交互式终端应用，让用户可以手动驱动状态模型。问题关注**业务逻辑、状态转换或数据形状**时使用它——这些东西纸面上看起来合理，只有跑过真实场景才会感觉哪里不对。

## 什么时候适合这个形态

- “我不确定这个状态机能不能处理先 X 再 Y 的边界情况。”
- “这个数据模型真的能表示这种情况吗……”
- “写 API 前，我想先感受一下它应该长什么样。”
- 任何用户想要**按按钮并观察状态变化**的场景。

如果问题是“这个应该长什么样”，就选错分支了。使用 [UI.md](UI.md)。

## 流程

### 1. 说明问题

写代码前，先写下你要验证哪个状态模型，以及原型要回答什么问题。用一段话写在原型的 README 中，或写在文件顶部注释里。回答错问题的逻辑原型纯属浪费；把问题写明确，方便之后检查，无论用户现在在线还是之后 AFK 回来看。

### 2. 选择语言

使用宿主项目已有语言。如果项目没有明显运行时（例如文档仓库），先问用户。

匹配项目既有工具约定；不要为了原型新增包管理器或运行时。

### 3. 把逻辑隔离到可移植模块里

把真正回答问题的逻辑放在一个小而纯的接口后面，这样它之后可以被提出来放进真实代码库。外层 TUI 是可丢弃的；逻辑模块不应该是。

合适形态取决于问题：

- **纯 reducer**：`(state, action) => state`。适合 action 是离散事件且 state 是单个值的场景。
- **状态机**：显式状态和转换。适合“当前到底哪些 action 合法”也是问题的一部分。
- **一组作用于普通数据类型的纯函数**。适合没有隐式当前状态、只是做转换的场景。
- **带清晰方法表面的 class 或 module**。适合逻辑确实拥有持续内部状态的场景。

选择最适合当前问题的形态，**不要**选择最容易接到 TUI 上的形态。保持纯净：无 I/O、无终端代码、不要用 `console.log` 做控制流。TUI 导入它并调用它；不要反向依赖。

这会让原型在自身生命周期之后仍然有价值。当问题被回答后，已验证的 reducer / machine / function set 可以提到真实模块中，TUI 外壳则删除。

### 4. 构建暴露状态的最小 TUI

把它做成一个**轻量 TUI**：每个 tick 清屏（`console.clear()` / `print("\033[2J\033[H")` / 等价方式）并重新渲染整帧。用户应该始终看到一个稳定视图，而不是不断增长的滚动日志。

每一帧按顺序包含两部分：

1. **当前状态**：美观打印，并便于 diff（每行一个字段，或格式化 JSON）。字段名或小节标题用**粗体**，较不重要的上下文（时间戳、ID、派生值）用**弱化显示**。可以直接使用原生 ANSI escape code：`\x1b[1m` 加粗，`\x1b[2m` 弱化，`\x1b[0m` 重置。除非项目已经有样式库，否则不需要引入。
2. **键盘快捷键**：列在底部，例如 `[a] 添加用户  [d] 删除用户  [t] 步进时钟  [q] 退出`。加粗按键、弱化说明，或反过来都可以，重点是读起来清楚。

行为：

1. **初始化状态**：单个内存对象/结构体。启动时渲染第一帧。
2. **每次读取一个按键（或一行）**，派发给会修改状态的 handler。
3. **每个动作后重新渲染**整帧；不要追加，替换。
4. **循环直到退出。**

整帧应该能放进一个屏幕。

### 5. 一个命令即可运行

把脚本加入项目现有任务运行器（`package.json` scripts、`Makefile`、`justfile`、`pyproject.toml`）。用户应该能运行 `pnpm run <prototype-name>` 或等价命令，不需要记路径。

如果宿主项目没有任务运行器，就把命令写在原型 README 顶部。

### 6. 交给用户

给用户运行命令。用户会自己驱动它；最有价值的时刻通常是他们说“等等，这不应该发生”或“嗯，我以为 X 会不一样”——这些就是**想法**里的 bug，也是原型的全部意义。如果他们想增加新动作，就加。原型本来就会演化。

### 7. 捕获答案

当原型完成使命后，问题的答案是唯一值得保留的东西。如果用户在线，问它教会了他们什么。如果不在线，就在原型旁边留下 `NOTES.md`，让答案在原型删除前可以被补上（或由你在观察过会话后补上）。

## 反模式

- **不要加测试。** 需要测试的原型就不再是原型（本条仅限原型代码，正式实现应使用 TDD）。
- **不要接真实数据库。** 除非问题明确就是持久化，否则使用内存存储。
- **不要泛化。** 不要写“以后如果要支持 X 怎么办”。原型只回答一个问题。
- **不要把逻辑和 TUI 混在一起。** 如果 reducer / state machine 引用了 `console.log`、prompt 或终端 escape code，它就不再可移植。让 TUI 做纯模块外层的薄壳。
- **不要把 TUI 外壳发到生产。** 外壳是为终端手动驱动优化的。它背后的逻辑模块才是值得保留的部分。
