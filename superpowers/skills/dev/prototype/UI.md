# UI 原型

在单一路由上生成**几个差异明显的 UI 变体**，并通过底部悬浮栏切换。用户在浏览器里切换变体，选一个（或从每个变体里各取一部分），然后丢弃其余内容。

如果问题关注逻辑/状态，而不是界面长什么样，就选错分支了。使用 [LOGIC.md](LOGIC.md)。

## 什么时候适合这个形态

- “这个页面应该长什么样？”
- “提交方案前，我想先看几个 dashboard 选项。”
- “给设置页试一个不同布局。”
- 任何用户本来要花一天在脑子里比较三个模糊 mockup 的场景。

## 两种子形态：强烈优先子形态 A

UI 原型贴着真实应用环境时更容易判断：真实 header、真实 sidebar、真实数据、真实信息密度。独立的可丢弃路由是个真空环境：每个变体孤立看都还行。只要有合理的既有页面能承载这些变体，默认使用子形态 A。只有原型确实没有附近归宿时，才使用子形态 B。

### 子形态 A：调整既有页面（优先）

路由已经存在。变体在**同一路由**渲染，并用 `?variant=` URL 查询参数控制。既有数据获取、参数和认证逻辑都保留；只替换渲染出来的子树。这是默认选择，除非有具体理由不这样做。

如果原型验证的东西还没有页面，但**自然会属于某个既有页面**（dashboard 的一个新区块、设置页的新卡片、既有流程里的新步骤），仍然算子形态 A。把变体挂载到宿主页面内部。

### 子形态 B：新页面（最后手段）

只有当被验证对象确实没有既有页面可以容纳时才使用，例如全新的顶层界面，或无法合理嵌入任何地方的流程。

按项目已有路由约定创建一个**可丢弃路由**；不要发明新的顶层结构。命名要明显表明它是原型，例如路径或文件名包含 `prototype`。同样使用 `?variant=` 模式。

决定使用子形态 B 前，先做 sanity check：真的没有既有页面能嵌入它吗？空路由会隐藏那些在真实、填充数据的页面里才会暴露的设计问题。

两种子形态都使用相同的底部悬浮切换栏。

## 流程

### 1. 说明问题并选择 N

默认生成 **3 个变体**。超过 5 个就不再是明显差异，而会变成噪音；最多 5 个。

在原型位置或文件顶部注释中，用一行写下计划：

> “设置页的三个变体，通过 `?variant=` 切换，挂在既有 `/settings` 路由上。”

无论用户是否在线，这都能留下清晰上下文。

### 2. 生成差异明显的变体

起草每个变体。每个变体都必须遵守：

- 页面目的和它能访问的数据。
- 项目的组件库 / 样式系统（TailwindCSS、shadcn、MUI、纯 CSS 等）。
- 清晰的导出组件名，例如 `VariantA`、`VariantB`、`VariantC`。

变体必须**结构上不同**：不同布局、不同信息层级、不同主要操作入口，而不只是颜色不同。三个轻微调整过的卡片网格不是 UI 原型，只是壁纸。如果两个草稿太相似，就用明确的“不要使用卡片网格”指导重做其中一个。

### 3. 把它们接起来

在路由上创建一个切换组件：

```tsx
// 伪代码 — 按项目实际框架调整
const variant = searchParams.get('variant') ?? 'A';
return (
  <>
    {variant === 'A' && <VariantA {...data} />}
    {variant === 'B' && <VariantB {...data} />}
    {variant === 'C' && <VariantC {...data} />}
    <PrototypeSwitcher variants={['A','B','C']} current={variant} />
  </>
);
```

子形态 A（既有页面）：保留切换器上方所有既有数据获取逻辑；只按变体替换渲染子树。

子形态 B（新页面）：在 `/prototype/<name>` 下创建可丢弃路由，挂载同一个切换器。

### 4. 构建悬浮切换器

在屏幕底部居中放一个小型固定定位栏，包含三部分：

- **左箭头**：切换到上一个变体（循环）。
- **变体标签**：显示当前变体 key；如果变体导出了名称，也显示名称。例如 `B — 侧边栏布局`。
- **右箭头**：切换到下一个变体（循环）。

行为：

- 点击箭头时更新 URL 查询参数（使用框架路由，例如 Next 的 `router.replace`、React Router 的 `navigate` 等），这样变体可分享且刷新后稳定。
- 键盘：`←` 和 `→` 方向键也可以切换。焦点在 `<input>`、`<textarea>` 或 `[contenteditable]` 内时，不要拦截方向键。
- 视觉上要和页面区分开（例如高对比 pill、轻微阴影），让人明显看出它不是正在评估的设计本身。
- 在生产构建中隐藏：用 `process.env.NODE_ENV !== 'production'` 或等价检查保护，避免意外合并原型时把切换栏发给用户。

把切换器放在单个共享组件中，这样两种子形态都能复用。位置遵循项目共享 UI 的组织方式。

### 5. 交给用户

给出 URL（以及 `?variant=` key）。用户有空时会自己切换。最有价值的反馈通常是**“我要 B 的 header 加 C 的 sidebar”**，这才是他们真正想要的设计。

### 6. 捕获答案并清理

某个变体胜出后，写下选择了哪个以及为什么（commit message、ADR、issue，或在 AFK 且用户尚未回复时写到原型旁边的 `NOTES.md`）。然后：

- **子形态 A**：删除失败变体和切换器；把胜出方案折入既有页面。
- **子形态 B**：把胜出变体提升为真实路由，删除可丢弃路由和切换器。

不要把变体组件或切换器留在仓库里。它们很快会腐烂，并且会迷惑下一个读代码的人。

## 反模式

- **变体只在颜色或文案上不同。** 那是微调，不是原型。真正的变体会在结构上产生分歧。
- **变体之间共享太多代码。** 共享 `<Header>` 可以；共享 `<Layout>` 会破坏原型意义。每个变体都应该能完全抛弃布局。
- **把变体接到真实写入操作。** 只读原型没问题。如果某个变体需要写入，指向桩实现；问题是“应该长什么样”，不是“后端能不能工作”。
- **直接把原型提升到生产。** 变体代码是在原型约束下写的（无测试、最少错误处理）。折入真实代码时要重写成生产质量。
