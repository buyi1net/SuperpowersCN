# 处理代码审查反馈

## 概览

代码审查需要技术判断，不需要情绪表演。

**核心原则：** 先验证，再实现。先问清楚，再假设。技术正确性优先于社交舒适感。

## 响应模式

```text
WHEN receiving code review feedback:

1. READ: 完整阅读反馈，不急着反应
2. UNDERSTAND: 用自己的话复述要求，或先提问
3. VERIFY: 对照代码库真实情况检查
4. EVALUATE: 判断它对当前代码库是否技术上成立
5. RESPOND: 技术性确认，或有理由地反驳
6. IMPLEMENT: 一次处理一项，每项都测试
```

## 禁止回应

**绝不要：**
- “你说得完全对！”（违反协作规范）
- “好建议！” / “非常好的反馈！”（表演式附和）
- “我现在就改”（尚未验证前）

**应该改为：**
- 复述技术要求
- 提出澄清问题
- 如果反馈错误，用技术理由反驳
- 直接开始工作，用行动代替客套

## 处理不清楚的反馈

```text
IF any item is unclear:
  STOP - 先不要实现任何内容
  ASK - 询问不清楚的条目

WHY: 条目之间可能有关联。理解不完整 = 实现错误。
```

**示例：**
```text
你的协作者：“修 1-6”
你理解 1、2、3、6，但不清楚 4、5。

错误：先实现 1、2、3、6，稍后再问 4、5
正确：“我理解 1、2、3、6。开始前需要先确认 4 和 5。”
```

## 按来源处理

### 来自你的协作者

- **可信**：理解后实现
- **范围不清楚时仍然要问**
- **不要表演式附和**
- **直接行动，或给出技术性确认**

### 来自外部审查者

```text
BEFORE implementing:
  1. Check: 对当前代码库是否技术上正确？
  2. Check: 是否会破坏现有功能？
  3. Check: 当前实现是否有历史原因？
  4. Check: 是否适用于所有目标平台 / 版本？
  5. Check: 审查者是否掌握完整上下文？

IF suggestion seems wrong:
  用技术理由反驳

IF can't easily verify:
  Say so: "我无法在没有 [X] 的情况下验证这一点。要我先 [调查/询问/继续] 吗？"

IF conflicts with your human partner's prior decisions:
  先停止，并与你的协作者讨论
```

**协作者规则：** “外部反馈要保持怀疑，但要仔细检查。”

## 对“专业化实现”的 YAGNI 检查

```text
IF reviewer suggests "implementing properly":
  grep codebase for actual usage

  IF unused: "这个端点没有被调用。是否直接移除（YAGNI）？"
  IF used: 再按实际使用场景完整实现
```

**协作者规则：** “你和 reviewer 都向我负责。如果不需要这个功能，就不要加。”

## 实现顺序

```text
FOR multi-item feedback:
  1. 先澄清所有不清楚的内容
  2. 再按以下顺序实现：
     - 阻塞问题（功能损坏、安全问题）
     - 简单修复（拼写、import）
     - 复杂修复（重构、逻辑）
  3. 每项修复单独测试
  4. 验证没有回归
```

多项反馈不要一口气批量改完。每项修改后尽量运行对应测试或最小验证，避免把新问题混进下一项。

## 什么时候反驳

以下情况要反驳：
- 建议会破坏现有功能
- 审查者缺少完整上下文
- 违反 YAGNI，例如建议实现未使用功能
- 对当前技术栈不正确
- 存在遗留兼容原因
- 与你的协作者已确定的架构决策冲突

**如何反驳：**
- 使用技术理由，不要防御性表达
- 提出具体问题
- 引用能工作的测试或代码
- 涉及架构决策时，让协作者参与

**如果不方便直接反驳：** 可以说：“这条反馈我需要先做技术核验。”

## 确认正确反馈

如果反馈确实正确：
```text
正确：“已修复。变更内容：[简要说明]”
正确：“已定位到 [具体问题]，修复在 [位置]。”
正确：[直接修复，并展示代码变化]

错误：“你说得完全对！”
错误：“好建议！”
错误：“感谢指出！”
错误：“谢谢你……”
错误：任何感谢式套话
```

**为什么不说感谢：** 行动本身就说明你听到了反馈。直接修复即可。

**如果你准备写“谢谢”：** 删掉，改为说明修复内容。

## 优雅修正自己的反驳

如果你反驳后发现自己错了：
```text
正确：“你是对的。我检查了 [X]，它确实会 [Y]。现在实现。”
正确：“已验证，你的判断正确。我之前理解错了，因为 [原因]。现在修复。”

错误：长篇道歉
错误：解释自己为什么会反驳
错误：过度说明
```

陈述修正事实，然后继续。

## 常见错误

| 错误 | 修正 |
|------|------|
| 表演式附和 | 陈述要求，或直接行动 |
| 盲目实现 | 先对照代码库验证 |
| 批量修改但不测试 | 一次一项，每项测试 |
| 默认审查者一定正确 | 检查是否会破坏现有行为 |
| 回避反驳 | 技术正确性优先 |
| 只实现看懂的部分 | 先澄清所有不清楚项 |
| 无法验证但继续做 | 说明限制，询问方向 |

## 真实示例

**表演式附和（坏）：**
```text
Reviewer: “移除遗留代码”
错误：“你说得完全对！我来删……”
```

**技术验证（好）：**
```text
Reviewer: “移除遗留代码”
正确：“检查后发现构建目标是 10.15+，这个 API 需要 13+。仍需要保留兼容逻辑。当前实现的 bundle ID 有误，是修这个问题，还是放弃 13 以下支持？”
```

**YAGNI（好）：**
```text
Reviewer: “实现完整的指标追踪，包含数据库、日期过滤、CSV 导出”
正确：“检索代码库后发现没有任何地方调用这个端点。是否直接移除（YAGNI）？还是有我没看到的调用方？”
```

**不清楚的条目（好）：**
```text
你的协作者：“修 1-6”
你理解 1、2、3、6，但不清楚 4、5。
正确：“我理解 1、2、3、6。实现前需要先确认 4 和 5。”
```

## GitHub 线程回复

回复 GitHub inline review comments 时，回复到对应 comment thread：

```text
gh api repos/{owner}/{repo}/pulls/{pr}/comments/{id}/replies
```

不要把 inline review comment 的回复发成顶层 PR comment。

如果项目使用 GitLab、Gitee、Coding.net、Jira、Linear 或自托管平台，按对应平台的线程机制处理；无法直接调用平台 API 时，让用户提供评论内容或手动粘贴链接，不要擅自配置认证或代理。

## 中国本土化注意事项

- 国内网络环境下，`gh`、`glab` 或平台 API 可能不可用；只提示用户按团队规范配置认证令牌或代理，不自动修改全局配置。
- 中文团队的 review、PR/MR 评论和 issue 回复默认使用中文；文件路径、命令、标签常量、API 路径和平台名保持原样。
- 如果审查意见来自飞书、语雀、企业微信、Jira 或其他外部文档系统，让用户把反馈文本贴到当前会话，再按本 skill 处理。
- 对批量删除、关闭 issue、强推、重写历史等破坏性建议，必须先列出影响范围并取得明确确认。

## 底线

**外部反馈是待评估的建议，不是必须执行的命令。**

验证。提问。然后实现。

不表演式附和。始终保持技术严谨。
