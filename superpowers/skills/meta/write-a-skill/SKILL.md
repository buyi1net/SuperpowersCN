# 编写 Skill

帮助用户创建可维护、可触发、可执行的 agent skill。先明确用途和触发边界，再写 `SKILL.md`；如果用户要求单文件，就不要强行拆分资源文件。

## 工作流程

1. **收集需求**：问清楚任务、领域、使用场景、触发词、输入输出、是否需要脚本、是否有参考材料。
2. **选择策略**：决定是忠实翻译、中文增强、英文原版加中文导读，还是新建 skill。
3. **确定边界**：写清这个 skill 做什么、不做什么、什么时候不应触发。
4. **起草 skill**：写出可直接审查的 `SKILL.md` 内容；需要脚本时再考虑 `scripts/`。
5. **自测触发逻辑**：用正向和反向场景测试 `description`。
6. **复核清单**：交付前按清单自检。
7. **交付审查**：给用户审查；不要覆盖源文档，除非用户明确指定。

## 策略选择

- **忠实翻译**：原 skill 是执行规则，且用户只想中文化阅读/使用。尽量保持行为等价。
- **中文增强**：用户想适配中文工作流、增加边界、示例或固定输出格式。允许改造，但要标明不是原版等价翻译。
- **英文原版 + 中文导读**：原 skill 行为敏感、容易翻译漂移，或需要保持上游同步。优先新增导读，不改执行文件。
- **新建 skill**：目标任务和已有 skill 明显不同，或触发条件、工作流、资源文件都独立。

## Skill 目录结构

```text
skill-name/
├── SKILL.md           # 主指令，必需
├── REFERENCE.md       # 详细参考资料，按需
├── EXAMPLES.md        # 示例，按需
└── scripts/           # 工具脚本，按需
    └── helper.js
```

资源文件是可选项。不要为了让目录结构看起来完整而创建额外文件；只有内容确实需要独立维护或用户明确同意拆分时才拆。

## SKILL.md 模板

```md


# Skill Name

## Quick start

[最小可用示例]

## Workflow

[分步骤工作流程]
```

中文增强版也可使用：

```md
description: [一句话说明能力]。触发条件：用户提及 [关键词/场景]。勿触发于 [易混淆场景]。
```

## Description 写法

`description` 是 agent 判断是否加载 skill 的主要依据。它会和其他已安装 skills 的描述一起出现，agent 会根据它判断是否触发。

要求：

- 最多 1024 字符。
- 使用第三人称。
- 第一句写能力。
- 第二句写触发条件。
- 必要时写不应触发的易混淆场景。
- 保留 skill name、slash command、文件名、路径、包名、API 名称、CLI 命令等英文标识。

中文填空模板：

```text
[能力说明]。触发条件：用户提及 [关键词1]、[关键词2]、[关键词3]。勿触发于 [易混淆场景]。
```

好例子：

```text
围绕计划或设计高强度追问用户，压力测试方案，直到达成共享理解。触发条件：用户提及 grill me、拷问我、追问我、帮我压力测试。勿触发于用户已经要求直接实现的场景。
```

坏例子：

```text
帮助用户处理设计。
```

坏例子太泛，agent 无法区分它和其他设计、计划或架构类 skills。

## 中英混排

英文术语作为名词嵌入中文句子，不要半句中文半句英文。

差：

```text
Use when 用户想 grilling 一个计划。
```

好：

```text
当用户想用 grilling 方式压力测试计划时使用。
```

差：

```text
帮用户 create 一个 skill with proper structure。
```

好：

```text
帮助用户创建结构清晰的 skill。
```

保留英文标识：skill name、slash command、文件名、路径、包名、API 名称、CLI 命令、代码符号。

## 自测触发逻辑

写完草稿后，先做 3 个自测：

1. 用 `description` 里的触发词模拟一句用户请求，看 agent 是否应选择这个 skill。
2. 检查它和其他已安装 skills 的 `description` 是否重叠。
3. 用“不应触发”的场景反向测试，看是否会误触发。

自测样例：

```text
我想写一个 skill，帮我把接口文档整理成 PRD。
```

应触发 `write-a-skill`。

```text
帮我用现有 /to-prd 生成一个 PRD。
```

不应触发 `write-a-skill`，因为用户要使用已有 skill，不是创建新 skill。

## 复核清单

- [ ] `description` 写清能力和触发条件。
- [ ] `description` 不超过 1024 字符。
- [ ] `name` 是稳定的 ASCII slug。
- [ ] 术语一致，中英混排自然。
- [ ] 示例具体，但不包含真实密钥、token、生产 URL 或危险命令。
- [ ] 明确不适用场景，避免抢占其他 skills。
- [ ] 若有脚本，脚本只处理确定性、重复性、低风险任务。
- [ ] 若有参考文件，主文件说明何时读取，不要求默认全读。
- [ ] 输出前确认文件名，不覆盖源文档，除非用户明确指定。

## 安全边界

不要在示例中写真实密钥、token、生产 URL 或真实用户数据。涉及删除、覆盖、推送、迁移、外部服务调用等操作时，必须要求用户明确确认。不要编造 URL、任务项、路径、权限或承诺 skill 无法保证的能力。

危险命令只可作为安全警告示例，且必须明确不要执行。不要把 destructive 操作伪装成普通示例。

## 资源文件与脚本

拆文件不是拆 skill。一个目录仍然只有一个入口 `SKILL.md`；`REFERENCE.md`、`EXAMPLES.md` 是资源文件。

如果用户要求单文件，优先保持单文件。只有满足以下情况且用户同意时才拆文件：示例很多、参考资料需要独立维护、高级内容低频使用、脚本或资源会显著提升可靠性。

只有当操作确定、重复、低风险，且脚本比自然语言指令更可靠时，才添加脚本。不要为了看起来完整而添加脚本；脚本会带来维护成本、权限风险和跨平台问题。

适合脚本：校验 frontmatter、检查链接是否存在、生成固定目录结构、格式化确定性输出。

不适合脚本：需要大量判断的设计决策、会修改外部系统的操作、删除/覆盖/迁移/推送等高风险操作。

## 输出方式

根据用户需求输出以下之一：

- 新 skill 的目录结构和文件内容。
- 对已有 skill 的修改建议。
- 不覆盖源文件的翻译版或增强版草稿。
- 需要用户确认的问题清单。

如果要写入文件，先检查目标目录现有文件，选择不冲突的新文件名；不要覆盖源文档，除非用户明确指定。
