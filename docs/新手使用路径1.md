# AI 编程 Skill 合成版工作流结构

> 当前导航稿。用于说明 `D:\skills\superpowers` 合成版中的 27 个 skill 如何按 AI 编程工作流使用。历史来源为 `skills-main` 与 `superpowers-main`。

## 总览

合成版按实际 AI 编程工作流组织，而不是按原项目目录组织。

```text
A. 开工准备
  -> B. 需求与设计
  -> C. 计划与拆解
  -> D. 实现
  -> E. 调试
  -> F. 质量与验证
  -> G. 代码审查
  -> H. 交付收尾

I. 持续改进 / J. 沟通辅助 / K. 元技能：贯穿全程，随时可用。
```

当前暂定总数：27 个 skill。

- `write-a-skill` 与 `writing-skills` 暂时都保留，等待人工审核后再决定是否合并或删除其一。
- `test-driven-development` 采用原 `superpowers-main` 版本，替换 `skills-main` 的 `tdd`。
- `systematic-debugging` 采用原 `superpowers-main` 版本，替换 `skills-main` 的 `diagnose`。
- `grill-with-docs` 暂定删除，仅保留 `grill-me` 与 `brainstorming`。
- `prototype` 与 `brainstorming` 同时保留，前者验证可运行原型，后者输出设计文档。

## A. 开工准备

开工准备类 skill 用于给仓库和会话建立基础规则。

| Skill | 来源 | 用途 | 适合什么时候用 |
|---|---|---|---|
| `setup` | `skills-main`，原 `setup-matt-pocock-skills` | 首次使用前配置 issue tracker、triage 标签、领域文档布局。 | 新项目第一次接入这套 skill 合集时。 |
| `using-superpowers` | 原 `superpowers-main` | 定义整体使用规则：每次响应前检查是否有 skill 匹配，用户指令优先于 skill 规则。 | 每次会话开始时自动加载或作为合集总则。 |

### 工作流说明

```text
新项目第一次使用
  -> 运行 setup
  -> 写入项目级配置
  -> 后续 skill 知道 issue tracker、标签、文档在哪里

每次新会话开始
  -> using-superpowers 生效
  -> agent 先检查当前任务是否匹配某个 skill
```

### 小白理解

- `setup` 像新手机开机向导：设置语言、网络、账号。
- `using-superpowers` 像交通规则：你不需要每次都读，但它一直约束行为。

## B. 需求与设计

需求与设计类 skill 用于防止一上来就写代码。

| Skill | 来源 | 用途 | 产出 |
|---|---|---|---|
| `grill-me` | `skills-main` | 对你的想法或方案进行高强度追问，直到需求和关键决策清楚。 | 口头共识、决策摘要。 |
| `brainstorming` | 原 `superpowers-main` | 通过逐轮澄清、方案对比和分段确认，将想法整理成正式设计。 | 设计文档。 |
| `prototype` | `skills-main` | 搭建一次性可运行原型，验证关键逻辑、状态机或 UI 方向。 | 可运行但可丢弃的原型代码。 |
| `to-prd` | `skills-main` | 把已讨论清楚的需求整理成 PRD，并可发布到 issue tracker。 | 产品需求文档。 |

### 工作流说明

```text
想法还模糊
  -> grill-me 追问
  -> 形成基本共识
  -> brainstorming 输出设计文档
  -> 如果设计里有不确定点，用 prototype 跑一版验证
  -> 验证后，用 to-prd 沉淀为正式需求文档
```

### 小白理解

- `grill-me`：像找人拷问你的想法，帮你暴露漏洞。
- `brainstorming`：像写施工图，先设计再动工。
- `prototype`：像试穿衣服，先跑起来看看合不合适。
- `to-prd`：像把口头讨论写成正式合同，后面不扯皮。

---

## C. 计划与拆解

计划与拆解类 skill 用于把大目标拆成可执行的小任务。

| Skill | 来源 | 用途 | 适合什么时候用 |
|---|---|---|---|
| `writing-plans` | 原 `superpowers-main` | 把设计或需求拆成 2-5 分钟一个的实现步骤，每步包含文件路径、代码意图和验证方式。 | 设计已批准，准备写代码前。 |
| `to-issues` | `skills-main` | 将 PRD 或计划拆成可独立承接的 issue，并发布到 issue tracker。 | 需要项目管理、排期、多人协作或 AFK agent 承接时。 |
| `triage` | `skills-main` | 管理 issue 状态：分类、补信息、标记 ready-for-agent、ready-for-human 或 wontfix。 | issue 列表变多，需要整理和派活时。 |

### 工作流说明

```text
设计文档已确认
  -> writing-plans 拆成实现步骤
  -> 如果需要进入项目管理流程，用 to-issues 拆成 issue
  -> 后续用 triage 维护 issue 状态
```

### 小白理解

- `writing-plans`：像把菜谱拆成每一步怎么做。
- `to-issues`：像把装修任务拆给水电工、木工、油漆工。
- `triage`：像快递分拣中心，决定每个包裹下一步去哪。

---

## D. 实现

实现类 skill 用于真正开始写代码，同时控制风险。

| Skill | 来源 | 用途 | 适合什么时候用 |
|---|---|---|---|
| `using-git-worktrees` | 原 `superpowers-main` | 创建隔离工作区，避免实验性修改污染主目录。 | 新功能或大改动开始前。 |
| `test-driven-development` | 原 `superpowers-main` | 强制红绿重构：先写失败测试，再写最小实现，再重构。 | 实现功能或修 bug 前。 |
| `executing-plans` | 原 `superpowers-main` | 按实现计划逐任务执行，带检查点。 | 想在当前会话内稳妥执行计划时。 |
| `subagent-driven-development` | 原 `superpowers-main` | 每个任务派一个全新子 agent 执行，并进行两阶段审查。 | 任务较多、相互独立，想加速时。 |
| `dispatching-parallel-agents` | 原 `superpowers-main` | 将多个完全独立的任务或故障并行分派给不同 agent。 | 多个模块或测试失败互不依赖时。 |

### 工作流说明

```text
准备实现
  -> using-git-worktrees 建隔离工作区
  -> test-driven-development 写失败测试
  -> 选择执行方式：
       方式 1：executing-plans 稳定逐步执行
       方式 2：subagent-driven-development 多 agent 执行
  -> 如果存在多个独立任务，用 dispatching-parallel-agents 并行推进
```

### 小白理解

- `using-git-worktrees`：像在沙盘里搭房子，搭坏不影响真房子。
- `test-driven-development`：像先画靶子再射箭。
- `executing-plans`：像照着菜谱一步步做。
- `subagent-driven-development`：像多个厨师分工做菜。
- `dispatching-parallel-agents`：像同时打多个客服电话处理不同问题。

## E. 调试

调试类 skill 用于出问题时避免瞎猜。

| Skill | 来源 | 用途 | 适合什么时候用 |
|---|---|---|---|
| `systematic-debugging` | 原 `superpowers-main` | 按根因调查、模式分析、假设验证、修复实施四阶段排查 bug。连续多次失败时停止盲修，重新审视架构或假设。 | 代码报错、测试失败、行为异常、性能回归时。 |

### 工作流说明

```text
发现 bug
  -> 读取错误信息
  -> 复现 bug
  -> 找工作中的相似代码做对比
  -> 提出单一假设
  -> 最小验证
  -> 精准修复
  -> 回归测试
```

### 小白理解

像医生看病：先问症状，再检查，再诊断，再开药。不能一上来乱开药。

---

## F. 质量与验证

质量与验证类 skill 用于防止“看起来好了，其实没好”。

| Skill | 来源 | 用途 | 适合什么时候用 |
|---|---|---|---|
| `verification-before-completion` | 原 `superpowers-main` | 声称完成前必须运行验证命令，并读取确认输出。没有证据不得说完成。 | 每次准备说“完成了”“修好了”前。 |
| `setup-pre-commit` | `skills-main` | 设置 Husky pre-commit：格式化、typecheck、测试。 | 希望提交前自动挡掉低级错误时。 |
| `git-guardrails-claude-code` | `skills-main` | 在 Claude Code 里拦截危险 git 命令，如 `push`、`reset --hard`、`clean`。 | 想防止 AI 或自己误执行破坏性 git 操作时。 |

### 工作流说明

```text
开发过程中
  -> setup-pre-commit 建立提交前质量门禁
  -> git-guardrails-claude-code 防止危险 git 操作

准备宣布完成
  -> verification-before-completion 运行验证
  -> 只有验证通过，才说完成
```

### 小白理解

- `verification-before-completion`：交作业前先自己对答案。
- `setup-pre-commit`：提交前自动检查错别字和基础错误。
- `git-guardrails-claude-code`：给危险按钮加保险盖。

## G. 代码审查

代码审查类 skill 用于发现实现缺陷，并正确处理反馈。

| Skill | 来源 | 用途 | 适合什么时候用 |
|---|---|---|---|
| `requesting-code-review` | 原 `superpowers-main` | 派子 agent 审查代码，按 Critical、Important、Minor 报告问题。 | 完成任务、重大功能或准备合并前。 |
| `receiving-code-review` | 原 `superpowers-main` | 收到审查意见后逐条验证，合理的改，不合理的反驳。 | 收到 AI 或人类 review 反馈后。 |

### 工作流说明

```text
完成一个任务
  -> requesting-code-review 发起审查
  -> 收到反馈
  -> receiving-code-review 逐条判断
  -> 修 Critical / Important
  -> 必要时再审查
```

### 小白理解

像写完作文让老师批改。但老师也可能错，你要看懂意见再改。

---

## H. 交付收尾

交付收尾类 skill 用于决定代码怎么进入主线。

| Skill | 来源 | 用途 | 适合什么时候用 |
|---|---|---|---|
| `finishing-a-development-branch` | 原 `superpowers-main` | 测试通过后，提供合并、创建 PR、保留分支、丢弃分支等选项，并带安全检查。 | 功能完成、验证通过后。 |

### 工作流说明

```text
所有实现完成
  -> verification-before-completion 验证通过
  -> requesting-code-review 审查通过
  -> finishing-a-development-branch 选择合并 / PR / 保留 / 丢弃
```

### 小白理解

像做完饭后决定：端上桌、打包、继续保温，还是倒掉。

---

## I. 持续改进

持续改进类 skill 不一定绑定某个功能开发，适合定期使用。

| Skill | 来源 | 用途 | 适合什么时候用 |
|---|---|---|---|
| `improve-codebase-architecture` | `skills-main` | 扫描代码库，寻找浅模块、紧耦合、难测试区域，并提出重构方案。 | 项目变乱、功能越来越难改、定期架构体检时。 |
| `zoom-out` | `skills-main` | 跳出局部代码，解释模块在整体系统中的位置、调用关系和影响范围。 | 看陌生代码、接手项目、准备改复杂模块前。 |

### 工作流说明

```text
接触陌生代码
  -> zoom-out 先看全局
  -> 明白模块位置后再动手

项目变复杂
  -> improve-codebase-architecture 做架构体检
  -> 找出值得重构的模块
```

### 小白理解

- `zoom-out`：先看地图，再看街道。
- `improve-codebase-architecture`：定期整理房间，防止越来越乱。

---

## J. 沟通辅助

沟通辅助类 skill 用于降低对话成本和延续上下文。

| Skill | 来源 | 用途 | 适合什么时候用 |
|---|---|---|---|
| `caveman` | `skills-main` | 极简沟通模式，删除废话，保留技术准确信息。 | AI 回答太长、需要省 token 或快速推进时。 |
| `handoff` | `skills-main` | 将当前对话压缩成交接文档，供新会话或新 agent 接手。 | 对话太长、AI 开始失忆、需要换会话时。 |

### 工作流说明

```text
AI 太啰嗦
  -> caveman

对话太长
  -> handoff 生成交接文档
  -> 开新会话读取交接文档
  -> 继续工作
```

### 小白理解

- `caveman`：把 AI 切成电报模式。
- `handoff`：像医生交班本，换人也能接着干。

## K. 元技能

元技能用于维护这套 skill 合集本身。

| Skill | 来源 | 用途 | 当前状态 |
|---|---|---|---|
| `write-a-skill` | `skills-main` | 创建、改造、汉化或增强 skill，强调中文本地化、安全边界和草稿流程。 | 暂时保留，等待和 `writing-skills` 对比。 |
| `writing-skills` | 原 `superpowers-main` | 用 TDD 方法创建 skill，强调先验证 agent 无 skill 时会失败，再写 skill 改变行为，并做 CSO 优化。 | 后续先出汉化草稿，人工审核后决定是否替换 `write-a-skill`。 |
| `using-superpowers` | 原 `superpowers-main` | 合集总规则，已在 A 类列出。 | 保留。 |

### 工作流说明

```text
要创建或改造 skill
  -> 先用 write-a-skill 处理中文本地化与边界
  -> 再对比 writing-skills 的 TDD / CSO 方法
  -> 审核后决定是否合并两者
```

### 小白理解

这类 skill 不是写业务代码的，是用来“训练和改造 AI 工作方法说明书”的。

## 推荐新手使用路径

```text
1. setup
   初始化仓库配置。

2. grill-me
   把模糊想法问清楚。

3. brainstorming
   形成正式设计文档。

4. prototype（可选）
   对不确定的逻辑或 UI 跑一个一次性原型。

5. writing-plans
   把设计拆成实现步骤。

6. using-git-worktrees
   创建隔离工作区。

7. test-driven-development
   先写测试，再写实现。

8. executing-plans 或 subagent-driven-development
   按计划执行。

9. systematic-debugging
   出 bug 时系统化排查。

10. verification-before-completion
    完成前跑验证。

11. requesting-code-review / receiving-code-review
    审查并处理反馈。

12. finishing-a-development-branch
    合并、PR、保留或丢弃。

辅助：
- caveman：嫌 AI 啰嗦时用。
- handoff：对话太长时用。
- zoom-out：看不懂代码时用。
- improve-codebase-architecture：定期做架构体检。
```
