# zh-cn-mode 与 skill 合集重构交接

日期：2026年05月26日

## 对话摘要

本轮主要围绕 `D:\SuperpowersCN\skills\meta\zh-cn-mode` 的运行逻辑、职责边界和后续重构方向展开。

已明确的核心判断：

- `zh-cn-mode` 当前不是普通翻译工具，而是“中文输出偏好基础设施”。
- 它当前承担了两类职责：本轮会话中文输出，以及跨 agent 写入全局语言偏好和安装 skill。
- 当前 `zh-cn-mode` 的强制语言规则过宽，尤其是“所有人类可读文本必须简体中文”“代码注释必须中文”等规则，一旦被其他 skill 前置引用，必然会和业务 skill、项目规范、文件既有风格发生冲突。
- `write-a-skill`、`writing-skills` 等 skill 内部也包含翻译、汉化、本地化和代码相关偏好，它们与 `zh-cn-mode` 的规则存在重合，不能简单通过“所有产出类 skill 都加载 zh-cn-mode”解决。
- 之前的翻译工作不能直接等同于可靠的本地化，需要重新设计 skill 合集的职责分层和冲突优先级。

## 已讨论出的职责分层

建议将 `zh-cn-mode` 降级为“默认中文偏好层”，而不是强制本地化引擎。

推荐优先级：

1. 用户明确要求。
2. 项目、仓库、文件或目标平台的具体规范。
3. 当前业务 skill 的专门规则。
4. `zh-cn-mode` 的默认中文偏好。

推荐定位：

- `zh-cn-mode`：默认与用户沟通使用简体中文；无更具体规则时，产物默认中文。
- `write-a-skill`：skill 汉化、增强、本地化审查和触发边界的权威规则。
- `writing-skills`：从零创建、编辑、验证 skill 的 TDD 流程层。
- 代码类 skill：代码注释、commit message、issue/PR 正文等应优先遵循项目规范、用户要求和当前任务目标。

## zh-cn-mode 建议改造方向

建议将 `SKILL.md` 改成薄路由层，不承载复杂业务逻辑。

建议入口：

1. 只读规则入口：只读取 `rules.md`，用于当前会话中文输出，无副作用。
2. 当前 agent 配置入口：只给当前 agent 写入最小全局语言偏好，并引用 `zh-cn-mode`。
3. 跨 agent 配置入口：询问用户是否把语言偏好和 `zh-cn-mode` 安装到其他 agent。

建议文件结构：

```text
skills/meta/zh-cn-mode/
├── SKILL.md
├── rules.md
├── setup-current-agent.md
├── setup-cross-agent.md
└── references/
    └── agent-paths.md
```

建议路由语义：

- “用中文回复我”“说中文”默认只读 `rules.md`。
- “以后默认用中文”“把当前 agent 设置成中文”走当前 agent 配置。
- “其它 agent 也配置”“全部 agent 都设置中文”走跨 agent 配置。
- 其他产出类 skill 前置加载时，只读 `rules.md`，不执行环境搭建。

## 安装与检测策略建议

不建议默认联网检查 agent 最新路径。

推荐检测优先级：

1. 当前运行环境事实。
2. 本地文件系统检测。
3. 本地配置文件解析。
4. CLI 只读自描述命令，例如 `--version`、`--help` 或配置查询。
5. `references/agent-paths.md` 的静态候选路径。
6. 用户确认。
7. 用户明确授权后的可选联网查证。

建议 `references/agent-paths.md` 从静态路径表升级为“检测策略表”，记录默认路径、配置文件、只读 CLI 检测方式、置信度规则和最后验证日期。

检测结果建议标记置信度：

- `confirmed`：本地文件、配置或 CLI 明确确认。
- `probable`：默认路径存在，但没有配置证据。
- `guessed`：只来自静态表。
- `unknown`：无法判断。
- `conflict`：检测结果互相矛盾。

只有 `confirmed` 和用户明确接受的 `probable` 应允许自动写入。`guessed` 和 `unknown` 只提示用户。

## 全局语言偏好建议

建议写入托管块，避免重复追加，也便于后续更新和卸载。

示例：

```md
<!-- zh-cn-mode:start -->
Always respond in Simplified Chinese (简体中文).
在输出文档、注释或任何人类可读文本前，加载一次 zh-cn-mode。
<!-- zh-cn-mode:end -->
```

注意：后续重构时应重新审查这段文案，避免它再次把 `zh-cn-mode` 变成强制本地化规则。更稳妥的措辞可能是“默认使用简体中文；如用户、项目规范或当前 skill 有更具体要求，以更具体要求为准”。

## 当前发现的具体风险点

- 当前 `setup.md` 的安装流程只提到复制 `SKILL.md`、`setup.md`、`rules.md`，但 `setup.md` 依赖 `references/agent-paths.md`，复制范围不完整。
- 当前 `<SUBAGENT-STOP>` 让子 agent 完全跳过 `zh-cn-mode`，如果子 agent 产出用户可读内容，可能绕开中文偏好。更合理的是子 agent 不执行环境搭建，但需要产出人类可读文本时仍可读取 `rules.md`。
- 当前“用户主动说用中文”容易被解释成批量配置所有 agent，副作用过大。应区分“本轮中文输出”“当前 agent 持久配置”“跨 agent 配置”。
- 当前 `rules.md` 对代码注释、文档、commit、issue/PR 的语言要求过硬，和其他 skill 及项目规范冲突概率高。
